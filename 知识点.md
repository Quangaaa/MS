### 一.1、**单例模式**：饿汉式、懒汉式（双重检查锁 `volatile` 禁止指令重排）、静态内部类、枚举单例的反序列化漏洞

volatile：‌**防止编译器优化**‌，多线程环境下保证变量的‌**可见性**‌和‌**有序性**‌，但‌**不保证原子性**。告知编译器该变量可能被外部因素（如硬件、中断、其他线程）修改，每次访问必须直接从内存读取，禁止使用寄存器缓存或省略读写操作‌；在 Java 等语言中，当一个线程修改 volatile 变量时，新值会立即刷新到主内存，其他线程读取时会强制从主内存加载最新值

| 写法                | 核心特点                                 | 线程安全性                    | 懒加载               | 防反序列化   |
| :------------------ | :--------------------------------------- | :---------------------------- | :------------------- | :----------- |
| **1. 饿汉式**       | 类加载时就创建实例                       | 安全（JVM保证类加载线程安全） | 否                   | 需额外处理   |
| **2. 懒汉式 (DCL)** | 首次调用才创建，双重检查锁+`volatile`    | 安全（需配合`volatile` ）     | 是                   | 需额外处理   |
| **3. 静态内部类**   | 利用类加载机制实现懒加载，代码简洁       | 安全                          | 是                   | 需额外处理   |
| **4. 枚举**         | 《Effective Java》推荐，自动防序列化破坏 | 安全                          | 否（但类加载即创建） | **天然免疫** |

#### 1. 饿汉式（Eager Initialization）

java

```
public class EagerSingleton {
    // 1. 类加载时直接 new 出来
    private static final EagerSingleton INSTANCE = new EagerSingleton(); //final确保 INSTANCE 引用不可被重新赋值，防止内部逻辑意外修改破坏单例
    
    // 2. 私有构造器
    private EagerSingleton() {}
    
    public static EagerSingleton getInstance() {
        return INSTANCE;
    }
}
```

- **原理**：JVM 的类加载机制（`<clinit>` 方法）是线程安全的，所以多个线程同时访问 `getInstance()` 不会产生多个实例。类加载机制本身是**同步**的（`ClassLoader.loadClass` 会加锁），因此不需要 `synchronized` 也能天然避免多线程并发创建多个实例的问题。
- **缺点**：如果这个对象很大，且可能一直用不到，就会造成**内存浪费**。
- **结合简历**：你做的**数据库连接池初始化**如果不需要懒加载，用饿汉式其实最稳妥，性能也最好（没有锁开销）。



clinit：编译器会将你在类中编写的**所有静态变量赋值语句**和**静态代码块 `static { }`**，按照从上到下的顺序收集并合并成一个 `<clinit>` 方法。

| 方法名     | 对应 Java 概念         | 调用时机                       | 是否线程安全         |
| :--------- | :--------------------- | :----------------------------- | :------------------- |
| `<clinit>` | 类构造器（静态初始化） | 类加载初始化阶段（只执行一次） | **是**（JVM 锁保证） |
| `<init>`   | 实例构造器（构造函数） | 每次 `new` 对象时              | 否（需自行处理）     |

#### 2. 懒汉式：双重检查锁（DCL）+ `volatile` （重中之重）

java

```
public class DCLSingleton {
    // volatile 两个作用：1.保证可见性 2.禁止指令重排
    private static volatile DCLSingleton instance;
    
    private DCLSingleton() {}
    
    public static DCLSingleton getInstance() {
        if (instance == null) {                     // 第一重检查（无锁，提升性能）
            synchronized (DCLSingleton.class) {      // 类锁
                if (instance == null) {             // 第二重检查（防止重复创建）
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```



**面试官追问 1：为什么需要两次判空？**

- **第一重判空**：为了**性能**。如果对象已经存在，大部分线程走到 `if (instance == null)` 发现不为空就直接返回了，根本**不会进入同步块拿锁**。
- **第二重判空**：为了**安全**。假设两个线程都通过了第一重判空进入排队，A 拿到锁创建完对象释放锁，B 接着拿到锁。如果没有第二重判空，B 会**再次创建**一个新对象，导致单例失效。（两个同时判完第一个空之后A先拿锁 创建完之后B已经在判过空之后了）

**面试官追问 2：`volatile` 在这里到底防止了什么？**
这涉及到 **指令重排** 问题。`instance = new DCLSingleton()` 在 JVM 层面分为三步：

1. 分配内存空间。
2. 执行构造方法，初始化对象。
3. 将 `instance` 引用指向内存空间。

CPU 为了执行效率，可能把 **步骤 2 和步骤 3 对调**（StoreLoad 屏障问题 **StoreLoad 屏障问题**直接指向了 DCL 中 `volatile` 关键字最底层的救场机制 **该屏障之前的所有 Store（写）操作，必须在该屏障之后的任何 Load（读）操作执行之前，对所有处理器可见。**）：

- 如果线程 A 执行了 1 -> 3，**还未执行 2** 时，`instance` 已经不为 `null` 了。
- 此时线程 B 进入第一重判空，发现 `instance != null`，直接返回了一个**半初始化状态的对象**，调用其方法时会崩溃。

**`volatile` 的作用**：在变量写操作后插入**内存屏障**，强制禁止步骤 2 和步骤 3 的重排序，保证对象完整初始化后才被其他线程看到。

#### 3. 静态内部类（Holder 模式）

java

```
public class InnerClassSingleton {
    private InnerClassSingleton() {} //私有化构造
    
    // 静态内部类只有在被调用时才会被加载 不同懒汉式写外面初始化时直接创建
    private static class Holder {
        private static final InnerClassSingleton INSTANCE = new InnerClassSingleton();
    }
    
    public static InnerClassSingleton getInstance() {
        return Holder.INSTANCE;   // 触发 Holder 类的加载
    }
}
```



- **原理**：利用了 JVM 的**类加载时机**。`Holder` 类只有在 `getInstance()` 被第一次调用时才会被加载，加载过程中 JVM 会保证线程安全。相比 DCL，**代码极简且无需 `volatile` 和锁**。
- **结合简历场景**：这是你简历中未明确写出但**极其优雅**的单例写法，建议在面试时主动提及，体现对 JVM 机制的深度理解。

java

public enum EnumSingleton {

#### 反序列化漏洞与修复（简历中明确提及，必考点）

#### 1. 漏洞演示

假设你写了一个普通的 `DCLSingleton` 类（实现了 `Serializable` 接口）。攻击者可以通过以下代码强行生成第二个对象：

java

```
// 1. 正常获取单例
DCLSingleton s1 = DCLSingleton.getInstance();

// 2. 序列化写入文件
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("singleton.obj"));
oos.writeObject(s1);
oos.close();

// 3. 从文件反序列化读出
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("singleton.obj"));
DCLSingleton s2 = (DCLSingleton) ois.readObject();

System.out.println(s1 == s2);  // 输出 false！单例被破坏！
```



#### 2. 漏洞原理

反序列化时，JVM 会**绕过构造方法**，直接从字节流中重新构建一个全新的对象，因此绕过了 `private` 构造器和 `getInstance` 的逻辑。

#### 3. 修复方案

**方案一：在普通类中添加 `readResolve()` 方法**

java

```
public class DCLSingleton implements Serializable {
    // ... 原有代码 ...
    
    // 反序列化时会自动调用这个方法，用返回值替换流中生成的新对象
    protected Object readResolve() {
        return getInstance();   // 返回内存中的唯一实例
    }
}
```



**方案二：直接使用枚举单例（最推荐）**
枚举类在序列化时只记录名字（`INSTANCE`），反序列化时通过 `Enum.valueOf()` 查找已有的枚举常量，**永远不会创建新对象**。

### 一.2 **工厂方法**：简单工厂 vs 工厂方法 vs 抽象工厂的区别

工厂模式家族中这三个概念极易混淆，尤其是**简单工厂**并不属于 GoF 23 种设计模式，它只是一种编程习惯。我们从代码结构、扩展性原则和适用场景三个维度来拆解它们的根本区别。

#### 1. 核心定义与结构对比

| 对比维度               | 简单工厂 (Simple Factory)                                | 工厂方法 (Factory Method)                                    | 抽象工厂 (Abstract Factory)                                  |
| :--------------------- | :------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **模式归类**           | 编程习惯 / 静态工具类                                    | **类创建模式** (GoF)                                         | **对象创建模式** (GoF)                                       |
| **核心思想**           | **一个工厂类**根据传入参数 switch/if-else 返回不同产品。 | **定义一个用于创建对象的接口**，让**子类**决定实例化哪一个类。 | 提供一个**接口**，用于创建**一系列相关或相互依赖的对象**，无需指定具体类。 |
| **参与者数量**         | **3 个**：Factory + Product + ConcreteProduct            | **4 个**：Creator + ConcreteCreator + Product + ConcreteProduct | **至少 5 个**：AbstractFactory + ConcreteFactory + AbstractProductA/B + ConcreteProductA/B |
| **产品族 vs 产品等级** | 单一产品等级                                             | 单一产品等级                                                 | **多个产品等级构成一个产品族**                               |

#### 2. 代码层面的直观区别

假设我们要开发一个**日志记录器**，支持输出到**本地文件**或**远程网络**。

#### 场景一：简单工厂 (一个工厂盖所有)

如果老板说"加个数据库日志"，你需要**修改工厂类的 switch-case**。

java

```
// 违反开闭原则：新增产品必须修改工厂类代码
public class LoggerFactory {
    public static Logger getLogger(String type) {
        if ("FILE".equals(type)) return new FileLogger();
        else if ("NET".equals(type)) return new NetLogger();
        else return null;
    }
}
```



#### 场景二：工厂方法 (每家分店自己做菜)

如果老板说"加个数据库日志"，你需要**新增一个 `DatabaseLoggerFactory` 类**，原有代码**不用动**。

java

```
// 符合开闭原则：扩展新子类，不修改原有工厂代码
public abstract class LoggerFactory {
    public abstract Logger createLogger(); // 工厂方法
}
public class FileLoggerFactory extends LoggerFactory {
    @Override public Logger createLogger() { return new FileLogger(); }
}
```



#### 场景三：抽象工厂 (成套的 UI 皮肤)

**抽象工厂解决的是"产品族"问题。** 假设不仅要写日志，还要记录**审计信息 (Audit)**。Windows 风格的日志和审计是一套，Mac 风格的是另一套。

java

```
// 抽象工厂：负责创建两个产品 (Logger 和 Auditor)
public interface SystemComponentFactory {
    Logger createLogger();
    Auditor createAuditor();
}
// 具体工厂：Windows 家族
public class WindowsFactory implements SystemComponentFactory {
    public Logger createLogger() { return new WindowsEventLogger(); }
    public Auditor createAuditor() { return new WindowsRegistryAuditor(); }
}
```



#### 3. 本质区别的深度拆解

#### 区别一：继承 vs 组合 (扩展维度的差异)

- **简单工厂**：只有 **1** 个工厂类。新增产品 = **修改类内部逻辑**。
- **工厂方法**：有 **N** 个工厂子类。新增产品 = **新增子类** (利用多态/继承)。
- **抽象工厂**：有 **N** 个工厂接口。新增产品等级 (比如加个 `Formatter` 接口) = **修改顶层接口及所有子类** (这是抽象工厂最大的软肋)。

#### 区别二：产品维度的差异 (最重要的一张图)

| 模式                    | 处理维度                     | 举例说明                                                     |
| :---------------------- | :--------------------------- | :----------------------------------------------------------- |
| **简单工厂 / 工厂方法** | **一维** (产品等级)          | 只管 `Logger` 这一种东西的不同实现 (File/Net)。              |
| **抽象工厂**            | **二维** (产品族 + 产品等级) | 既要管 `Logger` 又要管 `Auditor`，且它们必须**配套** (Win配Win，Mac配Mac)。 |

#### 区别三：`new` 关键字在哪里？

这是辨别模式的口诀：

- **简单工厂**：`new` 写在**工厂类的静态方法里**。
- **工厂方法**：`new` 写在**子类的 `createXXX` 方法里**。
- **抽象工厂**：`new` 写在**具体工厂类的多个 `create` 方法里**。

#### 4. 为什么面试总问它们的区别？（选型决策表）

| 你的业务场景                                                 | 推荐模式                     | 理由                                                         |
| :----------------------------------------------------------- | :--------------------------- | :----------------------------------------------------------- |
| 产品类型极少，且未来几乎不变                                 | **简单工厂**                 | 代码最简单，过度设计是浪费。                                 |
| 只有一个产品等级，但未来会频繁增加新品种                     | **工厂方法**                 | 完美符合开闭原则，增加新产品不碰老代码。                     |
| 系统需要一系列**必须配对使用**的对象（如数据库连接+命令对象） | **抽象工厂**                 | 强制约束了产品族的一致性，防止 SQL Server 命令跑在 Oracle 连接上。 |
| 产品等级**和**产品族**同时**需要扩展                         | **考虑原型模式或建造者模式** | 抽象工厂支持增加产品族容易，增加产品等级极其困难（需改接口）。 |



### 一.3 **策略模式**：`Comparator` 接口、消除 `if-else` 的实现方式。

策略模式是消除冗长 `if-else` 和 `switch-case` 逻辑的经典手段，也是 `Comparator` 接口存在的底层逻辑。它把**算法的定义**与**算法的使用**解耦，让代码符合开闭原则。

#### 1. 核心定义与结构

- **意图**：定义一系列算法，把它们一个个封装起来，并且使它们可以**相互替换**。策略模式让算法的变化独立于使用它的客户端。
- **参与者**：
  - **Strategy（策略接口）**：定义算法族公共接口。对应 `java.util.Comparator<T>`。
  - **ConcreteStrategy（具体策略）**：实现具体算法。如 `SortByAge`、`SortByName`。
  - **Context（上下文）**：持有策略引用，负责调用策略。对应 `Collections.sort(list, comparator)`。

#### 2. 为什么 `Comparator` 是策略模式的教科书实现？

我们在给 `List` 排序时，不需要修改 `Collections.sort()` 的源代码，只需传入不同的 `Comparator` 实现即可改变排序逻辑。

java

```
List<User> users = Arrays.asList(new User("A", 30), new User("B", 20));

// 策略 1：按年龄排序
Collections.sort(users, new Comparator<User>() {
    public int compare(User o1, User o2) { return o1.getAge() - o2.getAge(); }
});

// 策略 2：按名字排序 —— 完全不需要修改 sort 方法，只需更换策略实现
Collections.sort(users, new Comparator<User>() {
    public int compare(User o1, User o2) { return o1.getName().compareTo(o2.getName()); }
});
```



**此处映射关系：**

- **Context** = `Collections` 工具类
- **Strategy** = `Comparator` 接口
- **ConcreteStrategy** = 匿名内部类 / Lambda 表达式

#### 3. 策略模式如何消除 `if-else`？—— 场景对比

假设一个电商支付场景，支持微信、支付宝、银行卡三种支付方式。

#### ❌ 传统反模式（大量 if-else 堆砌）

java

```
public String pay(String type, BigDecimal amount) {
    if ("WECHAT".equals(type)) {
        return "微信支付：" + amount;
    } else if ("ALIPAY".equals(type)) {
        return "支付宝支付：" + amount;
    } else if ("CARD".equals(type)) {
        return "银行卡支付：" + amount;
    } else {
        throw new IllegalArgumentException("支付方式不支持");
    }
}
```



**弊端**：每新增一种支付方式（如数字人民币），必须修改核心支付逻辑（`pay` 方法），分支越来越臃肿，违反开闭原则。

#### ✅ 策略模式重构（消除条件判断）

**步骤 1：定义策略接口**

java

```
public interface PaymentStrategy {
    String pay(BigDecimal amount);
}
```



**步骤 2：实现具体策略（每个分支独立成类）**

java

```
public class WechatPayment implements PaymentStrategy {
    @Override public String pay(BigDecimal amount) { return "微信支付：" + amount; }
}
public class AlipayPayment implements PaymentStrategy {
    @Override public String pay(BigDecimal amount) { return "支付宝支付：" + amount; }
}
public class CardPayment implements PaymentStrategy {
    @Override public String pay(BigDecimal amount) { return "银行卡支付：" + amount; }
}
```



**步骤 3：上下文环境 + 策略工厂（彻底消灭 if-else）**
我们需要一个地方来**创建并持有**策略。通常结合**工厂模式**来屏蔽 `if-else`。

java

```
// 工厂类负责根据类型返回策略（这里的 if-else 是不可避免的，但被隔离在工厂内部）
public class PaymentStrategyFactory {
    private static final Map<String, PaymentStrategy> STRATEGY_MAP = new HashMap<>();
    static {
        STRATEGY_MAP.put("WECHAT", new WechatPayment());
        STRATEGY_MAP.put("ALIPAY", new AlipayPayment());
        STRATEGY_MAP.put("CARD", new CardPayment());
    }
    public static PaymentStrategy getStrategy(String type) {
        PaymentStrategy strategy = STRATEGY_MAP.get(type);
        if (strategy == null) throw new IllegalArgumentException("不支持");
        return strategy;
    }
}

// 客户端调用 —— 核心逻辑中完全没有 if-else 痕迹
public class PaymentService {
    public String pay(String type, BigDecimal amount) {
        PaymentStrategy strategy = PaymentStrategyFactory.getStrategy(type);
        return strategy.pay(amount); // 多态调用，消除分支
    }
}
```



#### 4. 进阶技巧：结合枚举的极致优雅实现

如果策略是有限集合，用枚举实现策略模式是 Java 中最优雅的消除 `if-else` 手段。

java

```
public enum PaymentEnum implements PaymentStrategy {
    WECHAT {
        @Override public String pay(BigDecimal amount) { return "微信支付：" + amount; }
    },
    ALIPAY {
        @Override public String pay(BigDecimal amount) { return "支付宝支付：" + amount; }
    },
    CARD {
        @Override public String pay(BigDecimal amount) { return "银行卡支付：" + amount; }
    };
}

// 调用时直接一行代码，连工厂类都省了
String result = PaymentEnum.valueOf("WECHAT").pay(new BigDecimal("100"));
```



#### 5. 策略模式 vs 工厂方法 vs 状态模式

很多人在消除 `if-else` 时会混淆这几个模式，区别如下：

| 模式         | 关注点             | 关键区别                                                     |
| :----------- | :----------------- | :----------------------------------------------------------- |
| **策略模式** | 算法的**互换**     | 策略由**调用者主动选择**（我想用微信还是支付宝）             |
| **工厂方法** | 对象的**创建**     | 工厂不关心行为，只负责把对象 `new` 出来                      |
| **状态模式** | 行为的**自动切换** | 状态由**内部状态机驱动**变化（比如订单从待支付变已支付），调用者**被动感知** |

#### 6. 总结：策略模式的本质优势

1. **封装变化**：将 `if-else` 的每个分支逻辑独立封装成一个类，符合单一职责原则。
2. **开闭原则**：新增策略只需**新增一个类**，无需修改原有的 `Context` 和 `Factory` 核心逻辑。
3. **复用性**：策略类可以在系统其他地方独立调用和测试。
4. **消除条件判断**：利用**多态**代替**条件分支**，提升代码可读性。

### 一.4 **观察者模式**：JDK 自带的 `Observer` 类 vs Guava `EventBus`

观察者模式的核心在于**解耦事件发布者与订阅者**。在 Java 生态中，JDK 提供了原生的 `Observer` 和 `Observable`，而 Guava 提供了更现代的 `EventBus`。二者虽然目的相同，但设计哲学、使用体验和灵活性天差地别。

### 1. JDK 自带 `Observer` / `Observable` （1.0 时代的产物）

这是 JDK 1.0 就存在的内置支持，位于 `java.util` 包下。

#### 基础用法示例

java

```
// 1. 被观察者：继承 Observable
class NewsPublisher extends Observable {
    public void publishNews(String news) {
        setChanged();        // 必须标记状态已改变
        notifyObservers(news); // 推送消息
    }
}

// 2. 观察者：实现 Observer 接口
class NewsReader implements Observer {
    private String name;
    public NewsReader(String name) { this.name = name; }
    @Override
    public void update(Observable o, Object arg) {
        System.out.println(name + " 收到新闻: " + arg);
    }
}

// 3. 调用
NewsPublisher publisher = new NewsPublisher();
publisher.addObserver(new NewsReader("张三"));
publisher.addObserver(new NewsReader("李四"));
publisher.publishNews("今日油价下调");
```



#### 痛点与缺陷

| 问题点                           | 具体描述                                                     | 为什么这是致命的？                                           |
| :------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **1. 类爆炸与侵入性强**          | `Observable` 是一个**类**，不是接口。Java 单继承，若业务类已有父类，则**无法成为被观察者**。 | 严重限制代码复用，逼迫你修改类层级结构。                     |
| **2. `setChanged()` 反人类设计** | 每次通知前必须调用 `setChanged()`，否则 `notifyObservers()` 直接返回，**静默失败**。 | 极易遗忘导致 bug，且不易排查。                               |
| **3. 同步通知导致阻塞**          | `notifyObservers()` 内部**循环同步调用**每个观察者的 `update` 方法。 | 若一个观察者处理慢（比如发邮件、写磁盘），整个发布线程被**卡死**。 |
| **4. 事件类型模糊**              | `update(Observable o, Object arg)` 参数是 `Object`，需要强制转型且无法区分不同类型事件。 | 所有订阅者收到同一份杂烩数据，需要在 `update` 里写大量 `if-else` 判断事件来源。 |
| **5. 不可控的异常传播**          | 观察者 `update` 中抛出异常会直接中断后续观察者的通知，且发布者无法捕获处理。 | 一个坏掉的订阅者搞瘫整个通知链路。                           |

### 2. Guava `EventBus` （现代解耦利器）

Guava 在设计上完全规避了上述痛点，基于**注解（Annotation）**和**反射**实现，让任意类的任意方法都能成为事件处理器。

#### 基础用法示例

java

```
import com.google.common.eventbus.EventBus;
import com.google.common.eventbus.Subscribe;

// 1. 事件类（POJO，无任何特殊要求）
class NewsEvent {
    private String content;
    public NewsEvent(String content) { this.content = content; }
    public String getContent() { return content; }
}

// 2. 订阅者：只是一个普通类，方法上加 @Subscribe 注解
class NewsReader {
    private String name;
    public NewsReader(String name) { this.name = name; }
    
    @Subscribe
    public void onNews(NewsEvent event) {
        System.out.println(name + " 收到新闻: " + event.getContent());
    }
}

// 3. 调用
EventBus eventBus = new EventBus();
eventBus.register(new NewsReader("张三"));
eventBus.register(new NewsReader("李四"));
eventBus.post(new NewsEvent("今日油价下调"));
```



#### Guava EventBus 的核心优势

| 对比维度       | JDK Observer              | Guava EventBus                                               |
| :------------- | :------------------------ | :----------------------------------------------------------- |
| **侵入性**     | 必须继承 `Observable` 类  | **零侵入**，任意 POJO 类 + `@Subscribe` 注解即可             |
| **事件区分**   | 只有一个 `Object` 参数    | 根据**方法参数类型**自动路由事件（`onNews(NewsEvent e)`）    |
| **多事件处理** | 难以区分，需 `instanceof` | 一个订阅者可有**多个**带 `@Subscribe` 的方法，处理不同事件   |
| **线程模型**   | 同步阻塞                  | 默认同步，但可轻松切换为**异步**：`new AsyncEventBus(executor)` |
| **异常处理**   | 异常抛向上层，中断后续    | 提供 `SubscriberExceptionHandler` 接口，异常被**隔离捕获**，不影响其他订阅者 |
| **父子事件**   | 不支持                    | **支持继承**：若 `SportNewsEvent extends NewsEvent`，订阅 `NewsEvent` 的方法也能收到子类事件 |

### 3. 进阶用法：异步 EventBus 与 异常处理

java

```
// 1. 异步 EventBus：通知在独立线程池执行，不阻塞发布者
ExecutorService executor = Executors.newFixedThreadPool(4);
EventBus asyncBus = new AsyncEventBus(executor, (exception, context) -> {
    // 2. 异常处理器：记录日志，不影响其他观察者
    System.err.println("订阅者 " + context.getSubscriber() + " 处理事件 " + context.getEvent() + " 时异常：" + exception);
});

asyncBus.register(new SlowSubscriber()); // 里面有 Thread.sleep()
asyncBus.post(new NewsEvent("异步消息"));   // 主线程立即返回
```



### 4. Guava EventBus 的潜在缺点（选型须知）

- **反射开销**：基于注解和反射查找订阅方法，性能略低于硬编码调用（但在绝大多数业务场景中可忽略不计）。
- **编译期无检查**：方法签名写错或忘记加 `@Subscribe` 注解，编译期不会报错，运行时收不到消息。
- **内存泄漏风险**：注册后忘记 `unregister`，EventBus 会一直持有该订阅者的强引用。**务必在对象销毁时调用 `unregister()`**（在 Android/Spring 生命周期中尤为重要）。

### 5. 总结对比：一张表看清选型决策

| 场景                                           | 推荐方案                      | 理由                                               |
| :--------------------------------------------- | :---------------------------- | :------------------------------------------------- |
| **遗留系统维护**，类结构固定，改动代价大       | **JDK Observer**              | 原生的，不引入新依赖，能跑就别动。                 |
| **新项目开发**，追求代码简洁、解耦、可测性     | **Guava EventBus**            | 零侵入、异步支持、异常隔离，开发体验大幅提升。     |
| **极度追求性能**，高并发核心链路（如交易撮合） | **自行实现 Listener 模式**    | 直接维护 `List<Listener>` 循环调用，免去反射开销。 |
| **更复杂的响应式流控需求**                     | **Reactive Streams / RxJava** | 需要背压（Backpressure）控制时，EventBus 不够用。  |

> **一句话总结：JDK Observer 是能用但不好用的“老古董”，Guava EventBus 是符合直觉、高内聚低耦合的“现代工具箱”。除非有严格的依赖限制，否则 `EventBus` 是观察者模式在 Java 世界中的首选落地方式。**



### 一.5 **线程生命周期**：NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。

Java 线程的生命周期由 `java.lang.Thread.State` 枚举严格定义，共 **6 种状态**。理解这些状态的转换条件是分析死锁、性能瓶颈和线程 hang 住问题的基本功。

### 1. 六种状态的定义与进入条件

| 状态 (Enum)       | 含义与特征                                                   | 典型触发方法 / 进入条件                                      | 退出条件                                                     |
| :---------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **NEW**           | 线程对象已创建，但 **`start()` 尚未调用**。仅是一个普通对象，未关联操作系统线程。 | `Thread t = new Thread();`                                   | 调用 `t.start()` 后进入 `RUNNABLE`。                         |
| **RUNNABLE**      | **正在 Java 虚拟机中执行**。**注意**：它涵盖了操作系统层面的 **Running** 和 **Ready** 两种状态。只要线程可被 JVM 调度运行即为此状态。 | 调用 `start()` 后自动进入。                                  | 1. 等待锁未拿到 → `BLOCKED`。<br>2. 调 `wait/sleep/join/LockSupport.park` → `WAITING/TIMED_WAITING`。<br>3. 执行完 `run()` → `TERMINATED`。 |
| **BLOCKED**       | **阻塞于锁竞争**。线程试图进入 `synchronized` 修饰的方法或代码块，但该锁已被其他线程持有。 | 进入 `synchronized` 时锁被占用。                             | 一旦成功获取锁，立即回到 `RUNNABLE`。                        |
| **WAITING**       | **无限期等待**。线程不会被分配 CPU 时间片，直到被其他线程**显式唤醒**。 | 1. `Object.wait()` (无超时)<br>2. `Thread.join()` (无超时)<br>3. `LockSupport.park()` | 1. `Object.notify()/notifyAll()`<br>2. 等待的线程结束<br>3. `LockSupport.unpark()` |
| **TIMED_WAITING** | **有限期等待**。无需显式唤醒，时间到了自动回到 `RUNNABLE`。  | 1. `Thread.sleep(ms)`<br>2. `Object.wait(timeout)`<br>3. `Thread.join(timeout)`<br>4. `LockSupport.parkNanos/Until()` | 1. 超时时间到<br>2. 提前被显式唤醒                           |
| **TERMINATED**    | **线程已终止**。`run()` 方法正常执行完毕，或因未捕获异常退出。 | `run()` 结束。                                               | **不可逆**，调用 `start()` 会抛 `IllegalThreadStateException`。 |

### 3. 最容易混淆的两组对比

#### 对比一：BLOCKED vs WAITING (面试高频陷阱)

| 比较点           | BLOCKED                                                      | WAITING                                                      |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **根本原因**     | 竞争 **`synchronized` 内置锁**失败，处于锁对象的**入口队列 (Entry Set)**。 | 主动调用了 `wait()`，**已持有锁但主动释放**，处于锁对象的**等待队列 (Wait Set)**。 |
| **CPU 视角**     | JVM 在等待**操作系统级别的互斥量**。                         | JVM 在等待**其他线程的特定信号**。                           |
| **唤醒方式**     | **自动唤醒**：只要锁被释放且被 JVM 调度选中拿到锁。          | **必须唤醒**：必须调用 `notify()/notifyAll()`。              |
| **线程转储表现** | `state: BLOCKED (on object monitor)`                         | `state: WAITING (on object monitor)`                         |

#### 对比二：WAITING vs TIMED_WAITING

这两者除了 **“有无超时参数”** 外，在 `jstack` 日志中的表现截然不同：
- **WAITING**：线程堆栈会显示 `waiting on <地址>`，**没有**时间提示，说明它在无限期等待。
- **TIMED_WAITING**：堆栈会明确显示 `waiting on <地址> for [time]`，甚至显示 `sleeping`。

### 5. 重要提醒：RUNNABLE 不代表一定在跑 CPU

这是 Java 线程状态与操作系统线程状态最大的认知偏差。

- **操作系统视角**：`Ready` (就绪队列等待调度) vs `Running` (正在 CPU 执行)。
- **Java 视角 (RUNNABLE)**：**包含上述两者**。

**排查意义**：当看到大量线程处于 `RUNNABLE` 且 CPU 飙高，说明真的在**忙计算**；但如果 CPU 很低但 `RUNNABLE` 线程极多，大概率是遇到了 **`Read/Write` 系统调用返回慢** 或 **网络 IO 阻塞**，由于未进入 JVM 定义的 `BLOCKED/WAITING` 临界区，JVM 只能上报为 `RUNNABLE`。此时需要结合 `top -H` 和 `strace` 排查操作系统层面的系统调用。

### 一.6 **`synchronized` 底层原理**：对象头 Mark Word、锁升级过程（偏向锁 -> 轻量级锁 -> 重量级锁）。

`synchronized` 在 JVM 层面的实现完全依赖于**对象头 (Object Header)** 中的 **Mark Word**，并通过一套复杂的**锁升级 (Lock Inflation)** 机制在性能与安全之间寻求平衡。

#### 1. 根基：对象头与 Mark Word 结构

在 HotSpot 虚拟机中，每个 Java 对象在内存中分为三部分：**对象头**、**实例数据**、**对齐填充**。
对象头中最关键的是 **Mark Word**（长度 32 位或 64 位，取决于虚拟机位数），它是一块**动态复用**的数据区域。

以 **64 位 JVM** 为例，Mark Word 在不同锁状态下的位分布如下：

| 锁状态                     | 标志位 (lock) | 存储的关键信息                                               |
| :------------------------- | :------------ | :----------------------------------------------------------- |
| **无锁 (Normal)**          | `001`         | 对象的 `hashCode` (如果没重写 `hashCode()` 且未调用)，分代年龄 (GC Age) |
| **偏向锁 (Biased)**        | `101`         | **持有锁的线程 ID** (54 位 Epoch + 2 位偏向时间戳 + 线程 ID) |
| **轻量级锁 (Lightweight)** | `00`          | **指向当前线程栈中 Lock Record 的指针** (62 位)              |
| **重量级锁 (Heavyweight)** | `10`          | **指向操作系统互斥量 (Mutex) 的指针** (62 位)                |
| **GC 标记**                | `11`          | 空，供垃圾回收时标记对象使用                                 |

> **注意**：JDK 15 起默认**禁用偏向锁** (`-XX:-UseBiasedLocking`)，因为现代高并发应用下维护偏向锁的成本常高于收益。但面试仍考察偏向锁逻辑。

#### 2. 锁升级的完整生命周期

锁升级是 **单向不可逆** 的过程：无锁 → 偏向锁 → 轻量级锁 → 重量级锁。

#### 第一阶段：偏向锁 (Biased Locking) —— 极度乐观，假设只有一人用

- **触发条件**：当一个线程访问同步块并获取锁时，会在**对象头**和**当前线程栈帧的 Lock Record** 里存储**偏向的线程 ID**。
- **运行机制**：
  1. 检查 Mark Word 是否为**可偏向状态** (标志位 `101` 且 Thread ID 为空)。
  2. 如果是，使用 **CAS 操作**将当前线程 ID 写入 Mark Word。
  3. 后续该线程再次进入同步块时，**只需检查 Thread ID 是否是自己**，无需任何同步操作（零开销）。
- **撤销 (Revocation)**：当**另一个线程**尝试竞争该锁时，偏向锁会被撤销。JVM 需要等待**全局安全点 (Safe Point)** 暂停原持有偏向锁的线程，检查其是否还在执行同步代码，若已退出则恢复无锁状态；若未退出则**升级为轻量级锁**。

#### 第二阶段：轻量级锁 (Lightweight Locking) —— 自旋搏一搏，单车变摩托

- **触发条件**：多个线程**交替执行**同步块（无实际竞争冲突），或偏向锁撤销后。
- **运行机制**：
  1. 线程在当前栈帧中开辟一块空间 **Lock Record**，并将 Mark Word 的**备份** (Displaced Mark Word) 存于此。
  2. 使用 **CAS 操作**尝试将对象头的 Mark Word **替换为指向 Lock Record 的指针**。
  3. **成功**：获取轻量级锁，标志位变 `00`。
  4. **失败**：说明有其他线程**同时竞争**，当前线程**自旋等待**（循环执行 CAS 或空转）。
- **膨胀 (Inflation)**：如果自旋超过一定次数 (JVM 自适应调整) 仍未获取锁，或者有**第三个线程**介入竞争，轻量级锁会**膨胀为重量级锁**。

#### 第三阶段：重量级锁 (Heavyweight Locking) —— 完全依赖操作系统

- **触发条件**：存在实际**长时间竞争**或**等待线程过多**。
- **运行机制**：
  1. JVM 向操作系统申请一个 **Mutex Lock** (互斥量)。
  2. 对象头 Mark Word 存储指向该 Mutex 的**指针**，标志位变 `10`。
  3. 未获取锁的线程**进入阻塞状态 (BLOCKED)**，由操作系统内核完成**线程挂起与唤醒**（涉及用户态与内核态切换，开销巨大）。

#### 3. 为什么要有锁升级？（设计的智慧）

| 锁类型       | 优点                        | 缺点                                 | 适用场景                                                  |
| :----------- | :-------------------------- | :----------------------------------- | :-------------------------------------------------------- |
| **偏向锁**   | **极低延迟** (几乎无锁开销) | 撤销需安全点停顿，多线程竞争下是累赘 | **线程私有的工具类** (如 Vector, StringBuffer 在单线程下) |
| **轻量级锁** | **避免内核切换**，响应快    | 空自旋消耗 CPU 资源                  | **执行速度快、并发度低的代码块**                          |
| **重量级锁** | **稳定，不耗 CPU**          | 吞吐量低，线程阻塞/唤醒开销大        | **执行时间长、竞争激烈的业务逻辑**                        |

#### 4. 关键面试细节补遗

#### 细节 1：调用 `hashCode()` 会锁升级吗？

**会。** 偏向锁的 Mark Word 里**没有位置存储 `hashCode`**。

- 如果一个对象已经处于**偏向锁状态**，此时调用了 `Object.hashCode()`，偏向锁会被**强制撤销**，升级为轻量级锁（或重量级锁），`hashCode` 值会存储在 Mark Word 空闲区域。
- 如果一个对象**先计算了 `hashCode`**，Mark Word 已被占用，该对象**永远无法进入偏向锁状态**，只能直接从轻量级锁开始。

#### 细节 2：`wait()` / `notify()` 对锁的影响

- **一旦调用 `wait()`，锁直接膨胀为重量级锁**。
- 因为 `wait` 依赖于操作系统的条件变量 (Condition Variable)，轻量级锁和偏向锁层面不支持等待队列机制。

#### 细节 3：JIT 编译器的锁消除 (Lock Elimination)

即使写了 `synchronized`，JVM 通过**逃逸分析**发现锁对象不会被其他线程访问（如局部变量），会在 JIT 编译时**直接抹掉同步代码**，这比偏向锁还彻底。这是 `StringBuffer` 在单线程下性能并不比 `StringBuilder` 差的根本原因。

### 一.7 **`volatile` 关键字**：保证可见性、禁止指令重排（内存屏障），**但不保证原子性**（需结合 CAS 或锁）

**`volatile` 的作用**：在变量写操作后插入**内存屏障**，强制禁止步骤 2 和步骤 3 的重排序，保证对象完整初始化后才被其他线程看到。`volatile` 是 Java 并发编程中最轻量级的同步机制，它被称为**轻量级 `synchronized`**，但它只解决了并发中的两个问题：**可见性**与**有序性**，刻意**不解决原子性**。

#### 1. 核心机制一：保证可见性 (Visibility)

#### 问题背景：JMM 内存模型与 CPU 缓存

现代 CPU 有多级缓存，每个线程会把共享变量拷贝到自己的工作内存（CPU 缓存/寄存器）中操作。如果不加干预，线程 A 修改了变量，线程 B **可能永远看不到**新值。

#### `volatile` 的解决逻辑

- **写屏障**：当写一个 `volatile` 变量时，JVM 会强制将**当前 CPU 缓存行 (Cache Line) 的数据刷回主内存**。
- **读屏障**：当读一个 `volatile` 变量时，JVM 会强制**废弃当前 CPU 缓存行，直接从主内存重新加载**。

**代码示例（经典状态标志位）：**

java

```
public class VolatileFlagExample {
    // 如果不加 volatile，Worker 线程可能永远停不下来（JIT 优化导致）
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        new Thread(() -> {
            while (running) { 
                // 不加 volatile，这里读取的可能是寄存器里的旧副本
            }
            System.out.println("Worker stopped.");
        }).start();

        Thread.sleep(1000);
        running = false; // 主线程修改，Worker 线程立即可见
    }
}
```



#### 2. 核心机制二：禁止指令重排 (Ordering)

#### 问题背景：编译器和 CPU 的乱序执行

为了提高性能，编译器和 CPU 会对**没有数据依赖**的指令进行重排序。在单线程下无感知，但在多线程下可能导致**对象半初始化逃逸**（如 DCL 单例问题）。

#### `volatile` 的解决逻辑

`volatile` 通过在指令序列中插入**内存屏障 (Memory Barrier)** 来禁止重排序。

| 屏障类型            | 指令示例         | 作用                                                         |
| :------------------ | :--------------- | :----------------------------------------------------------- |
| **StoreStore 屏障** | `写 volatile` 前 | 禁止 `volatile 写` 与之前的**普通写**重排                    |
| **StoreLoad 屏障**  | `写 volatile` 后 | 禁止 `volatile 写` 与之后的**普通读/写**重排（全能屏障，开销最大） |
| **LoadLoad 屏障**   | `读 volatile` 后 | 禁止 `volatile 读` 与之后的**普通读**重排                    |
| **LoadStore 屏障**  | `读 volatile` 后 | 禁止 `volatile 读` 与之后的**普通写**重排                    |

**JVM 层面重排序规则（JSR-133 内存模型）：**

> **Volatile 写**具有与**锁释放**相同的语义。
> **Volatile 读**具有与**锁获取**相同的语义。

#### 3. 致命短板：不保证原子性

#### 为什么 `i++` 加 `volatile` 也救不了？

`i++` 操作在字节码层面是三条指令：

1. `getfield` (读 i)
2. `iconst_1 / iadd` (加 1)
3. `putfield` (写回 i)

`volatile` 只能保证 **步骤 1 读取的值是最新的**，以及 **步骤 3 写回的值是立即可见的**。但**无法阻止两个线程在步骤 1 和步骤 3 之间交错执行**。

java

```
public class VolatileAtomicityFail {
    private static volatile int count = 0;

    public static void main(String[] args) throws InterruptedException {
        CountDownLatch latch = new CountDownLatch(20);
        for (int i = 0; i < 20; i++) {
            new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    count++; // 非原子操作，最终结果必小于 20000
                }
                latch.countDown();
            }).start();
        }
        latch.await();
        System.out.println("Expected: 20000, Actual: " + count); // 大概率 18xxx
    }
}
```



#### 4. 总结对比：什么时候用 `volatile`？什么时候必须用锁/CAS？

| 场景描述                                                     | `volatile` 是否胜任     | 推荐方案                                           |
| :----------------------------------------------------------- | :---------------------- | :------------------------------------------------- |
| **一个线程写，多个线程读** (状态标志、配置项)                | ✅ **胜任**              | `volatile boolean flag`                            |
| **多个线程读，极少量并发写，且写操作不依赖原值**             | ✅ **勉强胜任** (需谨慎) | `volatile`                                         |
| **多个线程并发写，且结果依赖当前值** (`i++`、`if(x) then set`) | ❌ **完全不行**          | **`synchronized` / `AtomicInteger` / `LongAdder`** |
| **需要保证内存可见性 + 禁止重排，但写操作复杂**              | ❌ 只能保证变量自身可见  | **`synchronized` / `Lock`**                        |
| **双重检查锁定 (DCL) 单例**                                  | ✅ **必须使用** (JDK5+)  | `private static volatile Singleton instance;`      |

#### 5. 进阶技巧：利用 `volatile` 的低开销实现“轻量级读写锁”

在某些极高并发读、极少写的场景，可以利用 `volatile` 配合**写时复制 (Copy on Write)** 技巧来避免锁争用。

java

```
public class CheapReadWriteLock {
    private volatile Map<String, String> config = new HashMap<>();

    // 读：无锁，直接读 volatile 引用
    public String get(String key) {
        return config.get(key);
    }

    // 写：锁住构建过程，然后一次性替换 volatile 引用
    public synchronized void update(String key, String value) {
        Map<String, String> newConfig = new HashMap<>(config);
        newConfig.put(key, value);
        config = newConfig; // 原子替换，读线程瞬间可见新 Map
    }
}
```



#### 6. 经典面试追问

**问：`volatile` 和 `synchronized` 在可见性上的区别？**

- **相同点**：都能保证可见性。
- **不同点**：
  1. `volatile` 仅保证变量**自身**修改可见；`synchronized` 保证**进入/退出同步块时**所有共享变量（不仅仅是锁对象）的可见性。
  2. `volatile` 不会引起线程**上下文切换**；`synchronized` 在竞争激烈时会导致线程阻塞。

**问：`AtomicInteger` 底层是怎么解决原子性的？**

- `AtomicInteger` 利用 **CAS (Compare And Swap)** 指令 + **`volatile`** 变量。
- `private volatile int value;` 保证可见性。
- `Unsafe.compareAndSwapInt()` 利用硬件原子指令保证**比较-修改-写回**的原子性。



### 一.8 **7 大参数**：核心线程数、最大线程数、空闲存活时间、时间单位、阻塞队列、线程工厂、拒绝策略。

Java 线程池 `ThreadPoolExecutor` 的精髓完全凝聚在其构造方法的 **7 个参数** 中。理解这 7 个参数的配合机制，是避免线上 OOM（内存溢出）或线程泄漏的关键。

#### 1. 参数全景图与源码映射

```java
public ThreadPoolExecutor(int corePoolSize,
                          int maximumPoolSize,
                          long keepAliveTime,
                          TimeUnit unit,
                          BlockingQueue<Runnable> workQueue,
                          ThreadFactory threadFactory,
                          RejectedExecutionHandler handler)
```

这 7 个参数共同决定了 **“任务来了，线程池怎么动”**。

#### 2. 逐一拆解：每个参数的真实含义与坑点

#### 参数 1 & 2：`corePoolSize` (核心线程数) & `maximumPoolSize` (最大线程数)

这两个参数定义了线程池的**伸缩边界**。

| 对比项       | `corePoolSize`                                               | `maximumPoolSize`                                            |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **含义**     | 线程池中**常驻存活**的线程数量（即使它们闲着）。             | 线程池中**允许同时存在的最大线程数量**。                     |
| **创建时机** | 任务提交时，若当前线程数 < `corePoolSize`，**无论有无空闲线程，都新建线程**。 | 仅当**队列已满**且当前线程数 < `maximumPoolSize` 时，才会创建**非核心线程**。 |
| **回收机制** | 默认不会被回收（除非设置 `allowCoreThreadTimeOut(true)`）。  | 空闲时间超过 `keepAliveTime` 后，自动销毁。                  |

> **关键规律**：线程池会优先使用**核心线程**，其次**塞队列**，最后才**开辟最大线程**。

#### 参数 3 & 4：`keepAliveTime` (空闲存活时间) & `TimeUnit` (时间单位)

- **作用对象**：仅对 **超出核心线程数的那部分线程**（即 `poolSize - corePoolSize` 这部分）生效。
- **行为**：当非核心线程空闲时间超过设定值，该线程会被终止回收，直到线程池收缩回 `corePoolSize`。
- **特殊用法**：若调用了 `allowCoreThreadTimeOut(true)`，核心线程空闲超时也会被回收，线程池可收缩至 0。

#### 参数 5：`BlockingQueue<Runnable> workQueue` (阻塞队列) —— 线程池逻辑的核心分水岭

队列的选择直接决定了 `maximumPoolSize` 是否有机会被用到。

| 队列类型                              | 特征                                                         | 对线程增长的影响                                             | 典型场景                                                     |
| :------------------------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`SynchronousQueue`** (同步移交队列) | **容量为 0**，不存储任务。任务提交后必须立即有线程来取，否则阻塞。 | 极易触发创建**非核心线程**，直到达到 `maximumPoolSize`。若依然无空闲线程，则触发拒绝策略。 | **`Executors.newCachedThreadPool()`** (线程数无界的根本原因) |
| **`LinkedBlockingQueue`** (无界队列)  | 默认容量 `Integer.MAX_VALUE`。                               | **`maximumPoolSize` 形同虚设**。因为队列永远不会满，线程数永远不会超过 `corePoolSize`。 | **`Executors.newFixedThreadPool()`** (有 OOM 风险)           |
| **`ArrayBlockingQueue`** (有界队列)   | **推荐使用**。需指定容量。                                   | 队列满时，系统才被迫创建非核心线程，能有效控制资源使用。     | **生产环境首选**，配合 `CallerRunsPolicy` 构建弹性系统。     |

#### 参数 6：`ThreadFactory` (线程工厂)

- **作用**：统一封装 `new Thread()` 的过程。
- **最佳实践**：**必须自定义**。用于给线程设置**有意义的名字**（便于排查 jstack）、设置**是否为守护线程**、设置**未捕获异常处理器**。
- **错误示范**：使用默认工厂，出问题时面对一堆 `pool-1-thread-1` 无法定位业务归属。

#### 参数 7：`RejectedExecutionHandler` (拒绝策略)

当**队列已满**且**线程数已达 `maximumPoolSize`** 时，新提交的任务如何处理？

| 策略实现类                | 行为描述                                                     | 适用场景                                                     |
| :------------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`AbortPolicy`** (默认)  | 直接抛出 **`RejectedExecutionException`** 异常。             | 必须严格保证任务不丢失，丢任务就是系统故障。                 |
| **`CallerRunsPolicy`**    | **谁提交谁执行**。由提交任务的线程（如主线程/MVC 线程）亲自执行 `run` 方法。 | **高并发降级方案**。既能减缓任务提交速度（主线程干活去了没法继续提交），又不丢任务。 |
| **`DiscardPolicy`**       | **直接丢弃新任务**，毫无声息。                               | 极少使用（通常用于日志采样、非关键数据上报）。               |
| **`DiscardOldestPolicy`** | 丢弃**队列头部最老的任务**（即将被执行的任务），然后重新提交当前任务。 | 希望获取最新数据，容忍丢弃历史积压任务（如实时行情）。       |

#### 3. 参数联动的经典逻辑推演（面试必问）

**问：线程池提交一个任务后，内部执行顺序是什么？**

假设参数为：`core=5`, `max=10`, `queue=ArrayBlockingQueue(20)`。

1. 前 **5** 个任务到达：**直接创建核心线程**处理 (`poolSize = 5`)。
2. 第 **6 ~ 25** 个任务到达（共 20 个）：核心线程全忙，任务进入 **`workQueue` 排队**。
3. 第 **26** 个任务到达：此时队列**已满** (20个)，`poolSize` (5) < `max` (10)，**创建第 6 个非核心线程**处理。
4. 第 **27 ~ 30** 个任务到达：继续创建非核心线程，直到 `poolSize = 10`。
5. 第 **31** 个任务到达：队列满，线程数达到最大值，**触发拒绝策略**。

#### 4. 最佳实践：如何科学设置这 7 个参数？（避坑指南）

| 参数           | 推荐配置策略                                                 |
| :------------- | :----------------------------------------------------------- |
| **核心线程数** | **CPU 密集型**：`Runtime.getRuntime().availableProcessors() + 1`。<br>**IO 密集型**：`核心数 * 2` 或 `核心数 / (1 - 阻塞系数)`。 |
| **最大线程数** | 通常与核心线程数**保持一致**（使用有界队列兜底），或略高于核心数。切忌设置 `Integer.MAX_VALUE`。 |
| **存活时间**   | IO 密集型建议设长一些（如 60s），避免频繁创建销毁线程；纯内存计算设短一些无妨。 |
| **阻塞队列**   | **严禁使用无界队列**。建议 `ArrayBlockingQueue` 或 `LinkedBlockingQueue` 指定容量（如 1000）。 |
| **线程工厂**   | 务必重命名：`new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build()` (Guava)。 |
| **拒绝策略**   | **首选 `CallerRunsPolicy`**。让系统慢下来，而不要直接崩溃或丢数据。 |

### 一.9  **任务执行流程**：核心线程 -> 队列排队 -> 最大线程 -> 拒绝策略。 **常见阻塞队列**：`ArrayBlockingQueue` (有界)、`LinkedBlockingQueue` (默认无界危险)、`SynchronousQueue` (无容量直接交接)。 **拒绝策略**：`AbortPolicy` (抛异常)、`CallerRunsPolicy` (调用者执行)、`DiscardPolicy` (丢弃)、`DiscardOldestPolicy` (丢弃队首)

这正是线程池内部运转的三大核心支柱。为了方便复习和记忆，我将这三个关联机制整合在一张逻辑卡片中。

#### 1. 任务执行流程：核心线程 -> 队列 -> 最大线程 -> 拒绝策略

这是 `ThreadPoolExecutor.execute()` 方法的**唯一不可变逻辑**。记住这张流转图即可应对所有场景分析：

```text
新任务提交
    │
    ▼
当前线程数 < 核心线程数 (corePoolSize) ？
    │
    ├── 是 ──▶ 创建新线程执行（即使是刚创建的空闲线程）
    │
    └── 否
        ▼
    任务能否成功入队 (workQueue.offer) ？
        │
        ├── 能 ──▶ 任务进入队列排队等待
        │
        └── 不能（队列已满）
            ▼
        当前线程数 < 最大线程数 (maxPoolSize) ？
            │
            ├── 是 ──▶ 创建新线程执行（非核心线程，存活时间受 keepAliveTime 限制）
            │
            └── 否
                ▼
            执行拒绝策略 (RejectedExecutionHandler)
```

> **关键误区纠正**：并非任务一来就先塞队列。只有**核心线程满员**时，才会开始往队列里放。

#### 2. 常见阻塞队列特性对比

| 队列类型                  | 核心特征                                                | 对线程数增长的影响                                       | 风险评估                                                     |
| :------------------------ | :------------------------------------------------------ | :------------------------------------------------------- | :----------------------------------------------------------- |
| **`ArrayBlockingQueue`**  | **有界**。数组结构，需显式指定容量。                    | **推荐使用**。队列满后才迫使线程数增至最大，资源可控。   | 容量设置不当易触发拒绝。                                     |
| **`LinkedBlockingQueue`** | 链表结构，**默认容量 `Integer.MAX_VALUE`** (近似无界)。 | **最大线程数参数几乎失效**。线程数永远不超过核心线程数。 | **⚠️ 高风险**：高并发下队列无限积压，导致**内存溢出 (OOM)**。 |
| **`SynchronousQueue`**    | **没有容量**，不存任务。生产者必须等到有消费者来取。    | 极易触发创建非核心线程，直到达到 `maxPoolSize`。         | 配合超大 `maxPoolSize` 会导致线程数爆炸。                    |

#### 3. 四种拒绝策略的行为与适用场景

| 策略类                    | 行为描述                                                 | 一句话使用场景                                               |
| :------------------------ | :------------------------------------------------------- | :----------------------------------------------------------- |
| **`AbortPolicy`**         | **抛出 `RejectedExecutionException` 异常**。             | **系统必须感知过载**。任务丢失是严重故障，需要立即告警介入。 |
| **`CallerRunsPolicy`**    | **谁提交谁执行**（由调用线程执行 `run` 方法）。          | **优雅降级**。让上游线程（如 Tomcat 线程）帮忙干活，**自然减缓任务提交速度**，防止雪崩。 |
| **`DiscardPolicy`**       | **直接丢弃新任务**，毫无声息。                           | 日志采样、非关键埋点数据上报。                               |
| **`DiscardOldestPolicy`** | **丢弃队列中最老的任务**（队首元素），腾出位置放新任务。 | 要求获取最新数据，容忍丢失旧数据（如实时股价推送）。         |

#### 4. 强制避坑提醒（阿里开发手册精髓）

在实际编码中，**严禁使用** `Executors` 工厂类的以下两个方法直接创建线程池：
- `Executors.newFixedThreadPool()` —— 因为用了无界 `LinkedBlockingQueue`。
- `Executors.newCachedThreadPool()` —— 因为 `maximumPoolSize` 是 `Integer.MAX_VALUE`。

**正确姿势**：必须使用 **`new ThreadPoolExecutor`** 显式指定这 7 个参数，且**队列必须是有界队列**。

# 并发编程最后整体从零开始介绍（弱点）

### 二.1 四大隔离级别：读未提交、读已提交、可重复读、串行化。各自解决的并发问题（脏读、不可重复读、幻读）。 
数据库事务的隔离级别本质上是**并发性能**与**数据一致性**之间的权衡。级别越低，并发能力越强，但数据异常问题越多。

#### 1. 三大并发问题定义（理解的基础）

在讲隔离级别之前，必须先明确到底在“防”什么：

| 问题 | 定义 | 生活化类比 |
| :--- | :--- | :--- |
| **脏读 (Dirty Read)** | 事务 A 读取到了事务 B **未提交**的修改数据。一旦 B 回滚，A 读到的就是**无效的脏数据**。 | 你看到朋友购物车里还没付款的东西，你以为他买了，结果他删了。 |
| **不可重复读 (Non-Repeatable Read)** | 事务 A 内两次读取**同一行**数据，期间事务 B **修改并提交**了该行数据，导致 A 两次读到的**内容不一致**。 | 你查余额 100 块，去取钱时发现只剩 50 块（别人转走了并已到账）。 |
| **幻读 (Phantom Read)** | 事务 A 内两次查询**满足条件的记录数**，期间事务 B **插入或删除**了符合该条件的记录并提交，导致 A 第二次读到**多出或少了几行**。 | 你查部门人数是 10 人，打印名单时发现变成了 11 人（新人入职手续办完了）。 |

---

#### 2. 四大隔离级别对比表（MySQL 视角）

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现原理简述 (InnoDB) |
| :--- | :---: | :---: | :---: | :--- |
| **READ UNCOMMITTED** (读未提交) | ❌ 存在 | ❌ 存在 | ❌ 存在 | 不加锁，直接读最新数据（几乎无并发控制）。 |
| **READ COMMITTED** (读已提交) | ✅ 解决 | ❌ 存在 | ❌ 存在 | **MVCC**：每次 `SELECT` 生成 **ReadView**，读最新提交的快照。**Oracle 默认级别**。 |
| **REPEATABLE READ** (可重复读) | ✅ 解决 | ✅ 解决 | ⚠️ **部分解决** | **MVCC**：事务开始时生成 **ReadView**，全事务复用。**MySQL 默认级别**。 |
| **SERIALIZABLE** (串行化) | ✅ 解决 | ✅ 解决 | ✅ 解决 | **加锁**：普通 `SELECT` 隐式转为 `SELECT ... FOR SHARE`，读写串行。 |

> **特别说明 MySQL 的幻读**：在 **RR 级别**下，通过 **MVCC (快照读)** 解决了普通 `SELECT` 的幻读；但若执行 **`SELECT ... FOR UPDATE` (当前读)**，且未命中唯一索引，依然可能发生幻读，需配合 **间隙锁 (Gap Lock)** 彻底解决。

---

#### 3. 各级别深入剖析与 SQL 验证

#### 级别一：READ UNCOMMITTED —— 最低防线
- **现象**：事务 A 能看见事务 B 还未提交的改动。
- **SQL 模拟**：
    ```sql
    -- 事务 B
    SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
    START TRANSACTION;
    UPDATE account SET balance = 200 WHERE id = 1;
    -- 未提交

    -- 事务 A (同样设置级别为 READ UNCOMMITTED)
    SELECT balance FROM account WHERE id = 1; -- 结果为 200 (脏读)
    ```
- **应用场景**：几乎**没有**生产环境会用，除非是只读的纯分析报表且允许较大误差。

#### 级别二：READ COMMITTED —— Oracle 的默认选择
- **特点**：**语句级**一致性快照。每次 `SELECT` 都获取最新已提交数据。
- **SQL 模拟**：
    ```sql
    -- 事务 A
    START TRANSACTION;
    SELECT balance FROM account WHERE id = 1; -- 结果 100

    -- 事务 B (提交修改)
    UPDATE account SET balance = 200 WHERE id = 1;
    COMMIT;

    -- 事务 A 再次查询
    SELECT balance FROM account WHERE id = 1; -- 结果 200 (发生不可重复读)
    ```
- **适用场景**：互联网高并发业务，追求**高吞吐**，容忍主从延迟类的不一致。

#### 级别三：REPEATABLE READ —— MySQL 的黄金标准
- **特点**：**事务级**一致性快照。第一次 `SELECT` 建立快照，后续都读这个历史版本。
- **核心机制 (MVCC)**：利用 `trx_id` 和 `undo log` 链条，根据事务启动时的活跃事务列表构建视图。
- **关于幻读的特殊情况**：
    - **快照读** (`SELECT` 不加锁)：**解决了幻读**。
    - **当前读** (`SELECT ... FOR UPDATE`)：若没有间隙锁，B 插入数据后，A 再锁读会看到新行。
    - **MySQL 的 Gap Lock 补丁**：在 RR 级别下，当查询使用**唯一索引精准命中**时退化为记录锁；否则加**间隙锁**防止插入，从而**完全解决幻读**。

#### 级别四：SERIALIZABLE —— 绝对安全，绝对慢
- **行为**：事务完全串行执行。
- **死锁风险**：极高。
- **适用场景**：银行转账核心扣款逻辑、财务对账，确保数据万无一失。

---

#### 4. 经典面试追问：不可重复读 vs 幻读

这是最容易混淆的一组概念，核心区别在于**修改 (Update/Delete)** vs **新增 (Insert)**。

| 对比维度 | 不可重复读 | 幻读 |
| :--- | :--- | :--- |
| **针对对象** | 数据库中**已存在**的行数据 (Row) | 查询结果的**集合范围** (Result Set) |
| **SQL 动作** | `UPDATE` / `DELETE` | `INSERT` |
| **读到的变化** | **内容变了** (100块变200块) | **数量变了** (10行变11行) |
| **解决手段** | **行锁** 或 **MVCC 快照** | **间隙锁 (Gap Lock)** 或 **串行化** |

#### 5. 选型建议

| 场景 | 推荐隔离级别 | 理由 |
| :--- | :--- | :--- |
| 普通 Web 业务、查询列表 | **READ COMMITTED** | 间隙锁少，死锁概率低，并发性能好。 |
| 金融账务、余额扣减 | **REPEATABLE READ** 或 **SERIALIZABLE** | 防止对同一笔钱重复扣款或出现账目不平幻读。 |
| 数据分析、离线跑批 | **READ UNCOMMITTED** (特殊场景) | 数据量大，追求极致速度，允许少量误差。 |

### 二.2 MVCC 原理：基于 ReadView 和 Undo Log 链的快照读
多版本并发控制（MVCC）是 InnoDB 实现**读已提交（RC）** 和**可重复读（RR）** 隔离级别的核心底层技术。它允许读操作**不加锁**，直接读取历史版本快照，从而在读多写少的场景中达到极高的并发性能。

#### 1. 核心组件：两大基础设施

MVCC 的实现依赖于隐藏字段、Undo Log 和 ReadView 三者的精密配合。

#### 组件一：记录的隐藏列（每行数据都有）
InnoDB 为每一行记录（聚簇索引记录）添加了三个隐藏字段：

| 字段名 | 长度 | 作用 |
| :--- | :--- | :--- |
| **`DB_TRX_ID`** | 6 字节 | **最近修改（插入/更新）本行的事务 ID**。 |
| **`DB_ROLL_PTR`** | 7 字节 | **回滚指针**，指向写入 **Undo Log** 中的上一个版本。 |
| **`DB_ROW_ID`** | 6 字节 | 隐藏主键（当用户未定义主键时自动生成）。 |

#### 组件二：Undo Log 版本链（历史数据档案馆）
当一条记录被修改时，InnoDB 会：
1. 将当前记录的**旧值**拷贝一份到 Undo Log 中。
2. 新记录的 `DB_TRX_ID` 更新为当前事务 ID。
3. 新记录的 `DB_ROLL_PTR` 指向刚才拷贝的 Undo Log 旧版本。

**多次更新形成一条单向链表**，链头是最新记录，链尾是最老记录。

```text
[最新记录] (trx_id=300, roll_ptr→) 
    ↓
[Undo Log 版本2] (trx_id=200, roll_ptr→) 
    ↓
[Undo Log 版本1] (trx_id=100, roll_ptr→) 
    ↓
[初始记录] (trx_id=50, roll_ptr=null)
```

#### 组件三：ReadView（可见性裁决书）
ReadView 是**一个事务在进行快照读时产生的“视图快照”**，它包含四个核心属性：
- `m_ids`：生成 ReadView 时，系统内**活跃（未提交）的事务 ID 列表**。
- `min_trx_id`：`m_ids` 中的最小值。
- `max_trx_id`：生成 ReadView 时，系统**下一个即将分配的事务 ID**（即全局事务 ID 计数器当前值）。
- `creator_trx_id`：生成该 ReadView 的事务自己的 ID。

#### 2. 可见性判断算法：记录的历史版本能被我看到吗？

遍历 Undo Log 版本链时，对于某个版本的记录（设其 `trx_id = tid`），ReadView 按照以下规则判定其**可见性**：

| 条件 | 判断结果 | 解释 |
| :--- | :--- | :--- |
| **`tid == creator_trx_id`** | ✅ **可见** | 这个版本是我自己修改的，当然能看见。 |
| **`tid < min_trx_id`** | ✅ **可见** | 修改该版本的事务在我创建 ReadView 前**已经提交**了。 |
| **`tid >= max_trx_id`** | ❌ **不可见** | 修改该版本的事务是在我创建 ReadView **之后才启动**的。 |
| **`min_trx_id <= tid < max_trx_id`** | ⚠️ **需判断**：若 `tid` 在 `m_ids` 活跃列表中 → **不可见**；若不在列表中 → **可见**。 |

**若当前版本不可见**，则通过 `DB_ROLL_PTR` 指针顺着 Undo Log 链条**向前追溯上一个版本**，重复上述判断，直到找到第一个可见的版本为止。

#### 3. RC 与 RR 的本质区别：ReadView 生成的时机

这是面试中区分 RC 和 RR 底层原理的关键题眼。

| 隔离级别 | ReadView 生成时机 | 可见性效果 |
| :--- | :--- | :--- |
| **READ COMMITTED (读已提交)** | **每次 `SELECT` 语句执行前，都生成一个全新的 ReadView**。 | 同一事务内两次查询，如果期间别人提交了，第二次能看见别人提交的内容（**不可重复读**）。 |
| **REPEATABLE READ (可重复读)** | **仅在事务中的第一个 `SELECT` 语句执行时生成一次 ReadView，后续该事务内的所有 `SELECT` 复用此 ReadView**。 | 无论别人是否提交，我看到的永远是事务开始时的数据快照（**可重复读**）。 |

#### 4. 一个贯穿始终的例子

假设数据原值 `c=0`，初始 `trx_id=50`。
- **T1**：事务 100 开启，更新 `c=1`，提交。
- **T2**：事务 200 开启，更新 `c=2`，未提交。
- **T3**：事务 300 开启，更新 `c=3`，未提交。
- **T4**：**你的事务（ID=250）开启，执行 `SELECT`**。

**此时的状态：**
- 最新记录：`c=3`，`trx_id=300`
- 活跃事务列表 `m_ids`：[200, 300] （注意你的 ID=250 也在系统里，但针对你生成的 ReadView，`creator_trx_id=250`）
- `min_trx_id` = 200
- `max_trx_id` = 301（下一个待分配 ID）

**可见性判定过程（对于 `c=3` 这条记录）：**
1. 版本 `c=3`：`trx_id=300`。
    - `300 >= max_trx_id(301)`？ 不成立。
    - `300` 在活跃列表 `m_ids` [200, 300] 中吗？**在**。→ ❌ **不可见**。
2. 顺着 `roll_ptr` 找上个版本 `c=2`：`trx_id=200`。
    - `200` 在活跃列表 `m_ids` 中吗？**在**。→ ❌ **不可见**。
3. 再顺着找上个版本 `c=1`：`trx_id=100`。
    - `100 < min_trx_id(200)`。→ ✅ **可见**。

**结论**：即便别人已经把数据改成了 3 和 2，你在 RR 级别下读到的依然是 **1**。

#### 5. 重要概念澄清：快照读 vs 当前读

| 类型 | 触发方式 | 是否使用 MVCC | 行为 |
| :--- | :--- | :--- | :--- |
| **快照读 (Snapshot Read)** | 普通 `SELECT * FROM table` | **是** | 读取符合 ReadView 可见性规则的历史版本，**不加锁**。 |
| **当前读 (Current Read)** | `SELECT ... FOR UPDATE`<br>`SELECT ... LOCK IN SHARE MODE`<br>`UPDATE` / `DELETE` / `INSERT` | **否** | 读取记录的**最新已提交版本**，并对其**加锁**（记录锁/间隙锁）。 |

MVCC 只负责**快照读**的无锁并发。一旦涉及写操作或显式加锁查询，就必须退回到基于锁的并发控制（LBCC）来保证数据正确性。

### 二.3 间隙锁 (Gap Lock)：仅在 RR 级别 且 当前读 (select ... for update / update) 时出现
间隙锁是 InnoDB 在**可重复读（RR）**隔离级别下，用来彻底解决**幻读（Phantom Read）** 的终极武器。它不锁定具体的行，而是**锁定索引记录之间的“空隙”**。

### 1. 核心前提：间隙锁的触发条件（缺一不可）

| 条件 | 具体要求 | 反例（不加间隙锁） |
| :--- | :--- | :--- |
| **1. 隔离级别** | **必须是 `REPEATABLE READ`** | `READ COMMITTED` 下无间隙锁，只有行锁。 |
| **2. 操作类型** | **当前读** (`SELECT ... FOR UPDATE` / `UPDATE` / `DELETE`) | **快照读** (`SELECT`) 只用 MVCC，不加任何锁。 |
| **3. 索引条件** | 查询条件**未命中唯一索引的精确定位**（非唯一索引、范围条件、等值但记录不存在） | 条件为 `WHERE id = 1` 且 `id` 是主键，此时退化为**记录锁**。 |

### 2. 三种锁形态的层级关系

InnoDB 的行锁在 RR 级别下实际上包含三种具体的锁实现：

| 锁类型 | 锁定范围 | 目的 |
| :--- | :--- | :--- |
| **记录锁 (Record Lock)** | 锁定**索引中具体存在的一行记录**。 | 防止该行被其他事务 `UPDATE` 或 `DELETE`。 |
| **间隙锁 (Gap Lock)** | 锁定**索引记录之间的空隙**（不含记录本身），是**左开右开**区间。 | 防止其他事务在空隙中 **`INSERT`** 新记录，从而防止幻读。 |
| **临键锁 (Next-Key Lock)** | **记录锁 + 该记录之前的间隙锁**，是**左开右闭**区间。 | InnoDB 在 RR 级别下的默认加锁算法，同时锁住记录和前间隙。 |

### 3. 间隙锁是如何工作的？（图文举例）

假设表 `user` 中只有 `age` 字段上的一个普通索引，现有数据 `age` 值为 **10, 20, 30**。

此时索引 B+Tree 逻辑上的间隙分布为：
```text
(负无穷, 10)  -- 间隙1
[10]          -- 记录
(10, 20)      -- 间隙2
[20]          -- 记录
(20, 30)      -- 间隙3
[30]          -- 记录
(30, 正无穷)  -- 间隙4
```

#### 场景：事务 A 执行当前读范围查询
```sql
-- 事务 A (RR 级别)
START TRANSACTION;
SELECT * FROM user WHERE age BETWEEN 15 AND 25 FOR UPDATE;
```
- 表中不存在 15 和 25，但满足该**范围条件**的记录只有 **age=20**。
- 此时 InnoDB 会施加 **Next-Key Lock**：
    - 锁定 `age=20` 的记录 (Record Lock)。
    - 锁定 `(10, 20)` 间隙 (Gap Lock)。
    - 锁定 `(20, 30)` 间隙 (Gap Lock)。

#### 并发测试：事务 B 尝试插入
```sql
-- 事务 B
INSERT INTO user (age) VALUES (11); -- 处于间隙 (10,20) -> ❌ 被阻塞
INSERT INTO user (age) VALUES (22); -- 处于间隙 (20,30) -> ❌ 被阻塞
INSERT INTO user (age) VALUES (31); -- 处于间隙 (30,正无穷) -> ✅ 成功 (未被锁定)
INSERT INTO user (age) VALUES (9);  -- 处于间隙 (负无穷,10) -> ✅ 成功
```

**效果**：事务 A 在第二次执行 `SELECT ... FOR UPDATE` 时，绝不会多出 `age=11` 或 `age=22` 的记录，**幻读被物理锁彻底消灭**。

### 4. 间隙锁的特殊性质与注意事项

#### 性质一：间隙锁之间不冲突
- 行锁 (Record Lock) 是**互斥的**（写写互斥）。
- **间隙锁 (Gap Lock) 仅仅是阻止插入，多个事务可以同时持有同一个间隙的间隙锁**。
- 例如：事务 A 锁住 `(10, 20)` 间隙，事务 B 也可以锁住 `(10, 20)` 间隙。它们都允许对方读，但谁都**不允许往里 `INSERT`**。

#### 性质二：等值查询但记录不存在时的退化
```sql
-- 假设表中没有 age=25
SELECT * FROM user WHERE age = 25 FOR UPDATE;
```
- 此时不会产生记录锁（因为没记录），而是产生一个**间隙锁 `(20, 30)`**。
- 这同样能防止别人插入 `age=25`，符合幻读防御要求。

#### 性质三：主键/唯一索引等值查询时的锁降级
```sql
-- id 是主键，记录存在
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```
- 此时**退化为 Record Lock**，不会产生 Gap Lock。
- 因为没有必要锁间隙（主键唯一，只要锁住这一行，就不可能有人再插入 `id=1`）。

### 5. 间隙锁的双刃剑：死锁风险

由于间隙锁允许并发持有，极易导致**死锁**。

**死锁重现：**
- **T1 (事务A)**：`SELECT * FROM user WHERE age = 20 FOR UPDATE;` (持有 age=20 的记录锁，以及间隙锁)
- **T2 (事务B)**：`SELECT * FROM user WHERE age = 21 FOR UPDATE;` (age=21 不存在，B 持有 (20, 30) 间隙锁)
- **T3 (事务A)**：`INSERT INTO user (age) VALUES (21);` —— A 等待 B 释放间隙锁。
- **T4 (事务B)**：`INSERT INTO user (age) VALUES (20);` (假设之前没20，或者更新20) —— B 等待 A 释放记录锁。

**死锁发生**。MySQL 检测到死锁后，会回滚其中一个事务以打破僵局。

### 6. 如何查看间隙锁？

```sql
-- 查看当前锁等待状态
SELECT * FROM performance_schema.data_locks\G;
```
在 `LOCK_MODE` 字段中：
- `X` / `S`：Next-Key Lock (记录+间隙)
- `X,REC_NOT_GAP` / `S,REC_NOT_GAP`：Record Lock (纯记录锁)
- `X,GAP` / `S,GAP`：Gap Lock (纯间隙锁)

### 7. 总结：一句话理清 MVCC、Record Lock、Gap Lock 的分工

| 并发问题 | MVCC (快照读) | 行锁 (Record Lock) | 间隙锁 (Gap Lock) |
| :--- | :---: | :---: | :---: |
| **脏读** | ✅ 解决 | - | - |
| **不可重复读** | ✅ 解决 | - | - |
| **幻读 (快照读)** | ✅ 解决 | - | - |
| **幻读 (当前读 / 写操作)** | ❌ 不管用 | ❌ 只管已有行 | ✅ **物理阻塞插入** |

> **最终口诀：RR 级别下，MVCC 管看，间隙锁管干（增删改）。**

### 二.4 掌握死锁场景（如两个事务互相插入间隙锁范围内的数据）。 记录锁、间隙锁、临键锁 (Next-Key Lock) 的区别。
在 InnoDB 的锁机制中，**记录锁、间隙锁、临键锁**是行级锁的三种具体形态。它们共同构成了 RR 隔离级别下防止幻读的物理防线，但间隙锁的“不互斥”特性也埋下了特殊的**死锁隐患**。

### 1. 三种锁的核心定义与区间表示

假设索引列有值 `10, 20, 30`。间隙分布为：(-∞, 10), (10, 20), (20, 30), (30, +∞)。

| 锁类型 | 锁定范围 | 区间表示 | 锁定对象 | 作用 |
| :--- | :--- | :--- | :--- | :--- |
| **记录锁 (Record Lock)** | **单个索引记录** | `[20]` | 精确命中且存在的行 | 阻止其他事务 **UPDATE/DELETE** 该行 |
| **间隙锁 (Gap Lock)** | **索引记录之间的空隙** | `(10, 20)` | 不存在的空隙区间 | 阻止其他事务在该区间 **INSERT** |
| **临键锁 (Next-Key Lock)** | **记录 + 该记录之前的间隙** | `(10, 20]` | 记录本身及其前驱间隙 | **RR 级别默认加锁算法**，同时锁行和锁间隙 |

> **记忆技巧**：
> - `(` 开区间：不包含该记录，只锁空隙。
> - `[` 闭区间：包含该记录，锁住行本身。
> - **临键锁 = 间隙锁 + 记录锁**（左开右闭）。

### 2. 间隙锁导致的经典死锁场景（两个事务互相插入）

这是 RR 级别下**最隐蔽的死锁**，因为它完全由**间隙锁的兼容性**引发，而非行锁冲突。

#### 假设前提
- 表 `t` 有普通索引列 `id`，当前数据：`id = 10, 30`（注意**缺少 20**）。
- 隔离级别：**REPEATABLE READ**。

#### 死锁复现时序

| 步骤 | 事务 A | 事务 B |
| :--- | :--- | :--- |
| **T1** | `BEGIN;`<br>`SELECT * FROM t WHERE id = 20 FOR UPDATE;` | `BEGIN;` |
| **T2** | （A 持有 **间隙锁 (10, 30)**，因为记录不存在） | `SELECT * FROM t WHERE id = 25 FOR UPDATE;` |
| **T3** | | （B 也持有 **间隙锁 (10, 30)**，**间隙锁不互斥**） |
| **T4** | `INSERT INTO t VALUES (20);` | |
| **T5** | **（A 等待 B 释放间隙锁）** ← **阻塞** | `INSERT INTO t VALUES (25);` |
| **T6** | | **（B 等待 A 释放间隙锁）** ← **阻塞** |

#### 死锁原理拆解
1. **T1/T2**：A 和 B 都执行了等值查询但记录不存在（`id=20` 和 `id=25` 都没有）。根据规则，InnoDB 都施加了**相同的间隙锁 (10, 30)**。
2. **间隙锁兼容性**：间隙锁的唯一目的是**防止插入**，它们之间**不互斥**。所以 A 和 B **同时成功持有了 (10, 30) 的间隙锁**。
3. **T4**：A 想要插入 `id=20`。插入操作需要获取**插入意向锁 (Insert Intention Lock)**，这是一种特殊的间隙锁，**必须等待该间隙上的其他间隙锁释放**。于是 A 阻塞，等待 B 释放。
4. **T5**：B 想要插入 `id=25`，同样需要插入意向锁，同样阻塞，等待 A 释放。
5. **死锁形成**：A 等 B，B 等 A。MySQL 检测到死锁，回滚其中一个事务（通常是回滚持有锁较少的事务）。

### 3. 三种锁的区别对比总结表

| 对比维度 | 记录锁 (Record Lock) | 间隙锁 (Gap Lock) | 临键锁 (Next-Key Lock) |
| :--- | :--- | :--- | :--- |
| **锁粒度** | 精确的**一行索引记录** | **索引记录之间的开区间** | **一个左开右闭区间** |
| **触发条件** | 1. 唯一索引等值命中<br>2. RC 级别所有当前读 | 1. RR 级别<br>2. 等值查询记录**不存在**<br>3. 范围查询的边界间隙 | RR 级别下**所有非精确命中的当前读**（默认行为） |
| **互斥关系** | **写写互斥** | **间隙锁之间不互斥**<br>（仅与插入意向锁互斥） | 记录锁部分互斥，间隙锁部分不互斥 |
| **防止的操作** | 防止该行被**修改或删除** | 防止在该间隙**插入新记录** | 既防改，又防插 |
| **在 RC 级别表现** | **存在** | **不存在**（RC 无间隙锁） | **不存在** |
| **死锁风险** | 经典资源竞争死锁 | **间隙锁死锁**（如上例） | 范围查询导致的复杂死锁 |

### 4. 如何从日志中识别这三种锁？

执行 `SELECT * FROM performance_schema.data_locks\G` 查看 `LOCK_MODE` 字段：

| LOCK_MODE 显示 | 对应锁类型 | 说明 |
| :--- | :--- | :--- |
| `X` 或 `S` | **临键锁 (Next-Key Lock)** | 默认显示，锁记录 + 前间隙 |
| `X,REC_NOT_GAP` | **记录锁 (Record Lock)** | 明确不含间隙 |
| `X,GAP` | **间隙锁 (Gap Lock)** | 只锁间隙，不锁记录 |
| `X,GAP,INSERT_INTENTION` | **插入意向锁** | 准备插入时等待间隙锁释放的状态 |

### 5. 避坑指南：如何减少间隙锁死锁？

1. **尽量使用 RC 隔离级别**：RC 不使用间隙锁，死锁概率大幅降低（代价是可能出现不可重复读和幻读，需业务逻辑兼容）。
2. **缩短事务长度**：间隙锁在事务提交后才释放，长事务持有间隙锁是死锁的温床。
3. **统一操作顺序**：如固定先插大 ID 后插小 ID，能减少互相等待的概率。
4. **唯一索引精确删除**：`DELETE FROM t WHERE unique_key = 1` 只会加记录锁，不加间隙锁。
5. **避免在 RR 级别下执行范围条件的当前读**，若必须执行，尽量缩小范围。

### 二.5 B+Tree 结构：非叶子节点存键值，叶子节点存数据且双向链表连接（范围查询优势）。
B+ 树是 InnoDB 存储引擎的**灵魂数据结构**。它的设计目标非常纯粹：**让磁盘 I/O 次数最少，让范围查询像坐滑梯一样顺滑**。

### 1. 核心结构：一棵严格分工的两层树

B+ 树将节点分为两类，各司其职：

| 节点类型 | 存储内容 | 职责 | InnoDB 中的对应概念 |
| :--- | :--- | :--- | :--- |
| **非叶子节点** (Non-leaf Node) | **仅存储键值 (Key) + 子节点指针** | **路标**。只负责导航，找到数据所在的大致方向。 | **非叶子页**，常驻内存 Buffer Pool。 |
| **叶子节点** (Leaf Node) | **存储完整数据行 (Data) + 键值** | **仓库**。存放实际数据，并通过**双向链表**串联。 | **叶子页**，存储聚簇索引记录或二级索引记录。 |

### 2. 三大设计特征解析

#### 特征一：非叶子节点只存键值，不存数据 —— 让“目录”极薄、极宽
- **现象**：假设键值是 8 字节的 `bigint`，指针是 6 字节。一个 16KB 的磁盘页（InnoDB 默认页大小）可以存储约 **1170 个索引项** (`16KB / (8+6)B ≈ 1170`)。
- **对比**：如果像 B 树那样在非叶子节点也存完整数据行（一行可能 1KB），一个页只能存 **16 行**。
- **效果**：同样高度的树，B+ 树的**扇出 (Fanout)** 大得多。**树高 3 层的 B+ 树，能支撑 `1170^3 ≈ 16 亿` 条数据**，而磁盘 I/O 只需 3 次。

#### 特征二：叶子节点存完整数据 —— 查什么都得走到底
- 无论查询主键还是二级索引，最终都必须遍历到**叶子节点**才能拿到数据。
- **稳定性**：所有查询的 I/O 次数相同（树的高度），性能非常**稳定**。

#### 特征三：叶子节点双向链表连接 —— 范围查询的“高速公路”
- 这是 B+ 树相比 B 树最大的**杀手锏**。
- 叶子节点之间通过**前驱指针 (Prev)** 和**后继指针 (Next)** 连接成一个**有序双向链表**。

### 3. 范围查询优势的直观演示

假设执行 `SELECT * FROM user WHERE age BETWEEN 20 AND 30;`

**B+ 树的执行路径：**
1. 在索引树上做**一次**查找，定位到 `age=20` 所在的叶子节点。
2. 顺着叶子节点的**链表指针 (Next)** 向右扫描，依次读取数据，直到遇到第一个 `age > 30` 的记录停止。

**优势对比表：**

| 对比项 | 普通 B 树 (非 B+) | B+ 树 (InnoDB) |
| :--- | :--- | :--- |
| **定位首条记录** | 1 次树查找 | 1 次树查找 |
| **读取第 2 ~ N 条记录** | **反复回退到父节点**寻找下一条（中序遍历），可能引发大量随机磁盘 I/O | **顺着链表指针直接读下一个叶子页**，通常是顺序 I/O |
| **I/O 模式** | 随机读为主，速度极慢 | 顺序读为主，充分利用磁盘预读 (Read-Ahead) |

### 4. 结合 InnoDB 两种索引的具体实现

#### 聚簇索引 (Clustered Index)
- **非叶子节点**：存主键值。
- **叶子节点**：存**完整的行数据** (包括所有列)。
- **链表作用**：主键范围查询极快，例如分页 `WHERE id > 1000 LIMIT 10`。

#### 二级索引 (Secondary Index)
- **非叶子节点**：存索引列的值。
- **叶子节点**：存**索引列的值 + 对应的主键 ID**。
- **链表作用**：覆盖索引 (Covering Index) 场景下，范围扫描只需遍历二级索引的叶子链表，**无需回表**，性能达到顶峰。

### 5. 总结：B+ 树为何是数据库的最终答案

| 设计点 | 解决的问题 | 带来的收益 |
| :--- | :--- | :--- |
| **非叶节点无数据** | 单页存更多指针，树高度降低 | **减少磁盘 I/O 次数** |
| **叶节点存全量数据** | 查询路径等长 | **性能稳定可预测** |
| **叶节点双向链表** | 中序遍历无需回溯父节点 | **范围查询由随机 I/O 变顺序 I/O**，速度提升数十倍 |
索引列补充：
**索引列的值** 就是你在创建索引时指定的那一列或多列（即索引键）中，**实际存储的数据内容**。

举一个具体的例子，假设有这样一张表：
```sql
CREATE TABLE user (
    id   INT PRIMARY KEY,      -- 主键列
    name VARCHAR(20),          -- 姓名列
    age  INT                   -- 年龄列
);

-- 在 name 列上创建一个二级索引
CREATE INDEX idx_name ON user(name);
```

现在表中插入三行数据：

| id (主键) | name (姓名) | age (年龄) |
| :--- | :--- | :--- |
| 1 | 'Alice' | 25 |
| 2 | 'Bob' | 30 |
| 3 | 'Charlie' | 28 |

### 1. 什么是 “索引列的值”？

对于 **`idx_name`** 这个索引：
- **索引列** 是 `name` 这一列。
- **索引列的值** 就是 `'Alice'`、`'Bob'`、`'Charlie'` 这些具体的字符串。

### 2. 这个值在 B+ 树的哪里？起什么作用？

在 `idx_name` 这个**二级索引的 B+ 树**中：

| B+ 树节点 | 存放的内容 | 举例（对于 name='Bob' 这一行） |
| :--- | :--- | :--- |
| **非叶子节点** | **索引列的值** (`name`) | 存放 `'Bob'` 这个**键值**，用来与查询条件对比，决定往左还是往右子树搜索。 |
| **叶子节点** | **索引列的值 (`name`) + 对应的主键值 (`id`)** | 存放 `'Bob'` 和 **`2`**。 |

### 3. 它与 B+ 树搜索的联动关系

当你执行查询 `SELECT * FROM user WHERE name = 'Bob';` 时：

1. **在二级索引 B+ 树搜索**：
   - 从根节点开始，反复比对 **“索引列的值”**（比较字符串大小：`'Bob'` vs `'Alice'` 等）。
   - 最终在叶子节点定位到 **“索引列的值 = 'Bob'”** 这条记录。
2. **获取回表钥匙**：
   - 在该叶子节点记录中，取出配套的 **主键值 `id=2`**。
3. **去聚簇索引回表**：
   - 拿着 `id=2` 去聚簇索引 B+ 树再搜一次，取出完整的行数据。

### 4. 总结对比：不同索引中的 “索引列的值”

| 索引类型 | “索引列” 指什么？ | “索引列的值” 存在哪？ | 例子 |
| :--- | :--- | :--- | :--- |
| **聚簇索引** | 主键列 (如 `id`) | 非叶子节点存主键值；叶子节点存**完整行** | 主键值 `1`、`2`、`3` |
| **二级索引** | 你指定的列 (如 `name`) | 非叶子节点存该列值；叶子节点存 **该列值 + 主键值** | 姓名值 `'Alice'`、`'Bob'`、`'Charlie'` |

**一句话概括：**
> **索引列的值就是 B+ 树节点里存的那个用于比较大小、决定路线方向的“关键词”，也是在叶子节点里存着的实际数据片段。**

### 二.6 聚集索引 vs 二级索引：回表过程与索引覆盖原理
聚簇索引与二级索引的核心区别在于**叶子节点存放的内容不同**，这直接决定了查询是否需要**回表**以及能否利用**索引覆盖**优化。

### 1. 存储内容差异决定查询路径

| 索引类型 | 叶子节点存储内容 | 能直接提供的数据 |
| :--- | :--- | :--- |
| **聚簇索引** | **完整行数据**（所有列） | 任意列的值 |
| **二级索引** | **索引列的值 + 主键 ID** | 仅索引列的值和主键值 |

### 2. 回表过程：为什么二级索引要多跑一趟？

假设表结构：
```sql
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(20),
    age INT,
    INDEX idx_name (name)
);
```

执行查询：
```sql
SELECT * FROM user WHERE name = 'Bob';
```

**执行路径（两次 B+ 树查找）：**

1. **第一步：在二级索引 `idx_name` 的 B+ 树中查找**
   - 定位到 `name='Bob'` 的叶子节点，得到 `('Bob', id=2)`。
   - 二级索引到此为止，无法提供 `age` 列的数据。

2. **第二步：回表到聚簇索引**
   - 用得到的 `id=2` 作为键值，去聚簇索引的 B+ 树再做一次精确查找。
   - 在聚簇索引叶子节点获取完整行 `(2, 'Bob', 30)`，返回结果。

> **回表的代价**：多一次 B+ 树搜索，增加一次磁盘 I/O（若数据不在内存），且访问的页可能更分散。

### 3. 索引覆盖原理：让二级索引“独当一面”

如果查询只涉及**索引列**和**主键列**，二级索引的叶子节点已经包含了所需全部数据，**无需回表**。

```sql
-- 查询列只有 name 和 id，两者都在 idx_name 叶子节点中
SELECT id, name FROM user WHERE name = 'Bob';
```

**执行路径（一次 B+ 树查找）：**
- 在 `idx_name` 的 B+ 树定位到 `('Bob', id=2)`，直接从叶子节点取出 `id` 和 `name` 返回。
- **不回表**，I/O 减半，性能显著提升。

这种查询称为**覆盖索引 (Covering Index)**。

### 4. 进阶：联合索引的覆盖优化

联合索引将多个列组合成一个二级索引，叶子节点存储**所有索引列的值 + 主键**。通过合理设计联合索引，可以让更多查询享受覆盖索引的收益。

```sql
-- 创建联合索引 (name, age)
CREATE INDEX idx_name_age ON user(name, age);

-- 查询需要的列全部在联合索引叶子节点中 → 覆盖索引，不回表
SELECT name, age, id FROM user WHERE name = 'Bob' AND age = 30;
```

### 5. 一张图理清三者关系

```text
【查询 SELECT id, name, age FROM user WHERE name = 'Bob'】
                    │
                    ▼
        ┌───────────────────────┐
        │  二级索引 idx_name    │  （叶子节点只有 name 和 id）
        │  定位到 name='Bob'    │
        │  得到 id=2            │
        └───────────┬───────────┘
                    │
         （由于需要 age 列，必须回表）
                    ▼
        ┌───────────────────────┐
        │  聚簇索引 (主键 id)   │  （叶子节点有完整行）
        │  用 id=2 查找         │
        │  得到 (2, 'Bob', 30)  │
        └───────────────────────┘
                    │
                    ▼
              返回完整结果
```

### 6. 关键结论

| 场景 | 是否回表 | 原因 |
| :--- | :---: | :--- |
| `SELECT * FROM user WHERE name='Bob'` | ✅ **回表** | 二级索引无 `age` 列 |
| `SELECT id, name FROM user WHERE name='Bob'` | ❌ **不回表** | 所需列全在二级索引叶子节点 |
| `SELECT id FROM user WHERE name='Bob'` | ❌ **不回表** | `id` 本身就在叶子节点 |
| `SELECT age FROM user WHERE name='Bob'` | ✅ **回表** | `age` 不在二级索引中 |

> **一句话总结**：二级索引的叶子节点是“索引列+主键”的键值对，查不到的数据就得拿主键去聚簇索引“回炉重造”；如果能查全，就实现了覆盖索引的“就地解决”。

### 二.7 最左前缀法则：联合索引 (a,b,c) 在 where a=? and c=? 时的命中情况（会部分失效，仅用到 a）
### 1. 直接结论：仅命中 `a` 列

对于联合索引 `(a, b, c)`，查询条件 `WHERE a = ? AND c = ?`：

| 索引列 | 是否被用于**索引查找** | 原因 |
| :--- | :---: | :--- |
| `a` | ✅ **是** | 最左前缀的第一列，等值匹配，完美命中 |
| `b` | ❌ **否** | 查询条件**跳过了 `b`**，导致索引在 `a` 之后无法继续匹配 `c` |
| `c` | ❌ **否** | 被跳过的列 `b` 阻断，`c` 无法作为索引过滤条件使用 |

**最终效果**：索引树仅能通过 `a` 缩小扫描范围，然后在这个范围内**遍历所有 `b` 值**，再用条件 `c = ?` 逐行过滤。

### 2. 原理剖析：为什么跳过的列会让后续列失效？

联合索引的 B+ 树按照 **`(a, b, c)` 的字典序**组织叶子节点：

```text
索引值排序逻辑：
先按 a 排序 → a 相同则按 b 排序 → b 相同则按 c 排序
```

当查询只有 `a = ? AND c = ?` 时：
- 通过 `a = ?` 可以快速定位到第一棵以该 `a` 值为根的子树。
- 但在该子树内部，数据是**按 `b` 排序，再按 `c` 排序**的。
- 由于没有给定 `b` 的值，所有 `c = ?` 的记录会**散落在不同的 `b` 值区间内**，无法利用索引的有序性连续扫描，只能遍历该 `a` 值下所有的 `b` 和 `c`，再用 `c = ?` 筛选。

### 3. 特殊情况：并非“完全无用”——覆盖索引与 ICP 的救场

虽然 `c` 不能用于缩小索引扫描范围，但在以下两种优化机制下，索引依然能发挥部分作用。

#### 情况一：覆盖索引 (Covering Index)
若查询只涉及索引中的列，`EXPLAIN` 的 `Extra` 会显示 **`Using index`**。

```sql
-- 假设查询只需要 a, b, c 三列（或主键）
SELECT a, b, c FROM t WHERE a = 1 AND c = 3;
```
- **行为**：遍历 `a = 1` 的所有叶子节点（范围扫描），但**不需要回表**，直接从索引叶子节点取出 `c` 的值进行过滤。虽然扫描范围没变小，但省去了回表的 I/O 开销。

#### 情况二：索引条件下推 (Index Condition Pushdown, ICP)
MySQL 5.6+ 的优化。`EXPLAIN` 的 `Extra` 会显示 **`Using index condition`**。

```sql
SELECT * FROM t WHERE a = 1 AND c = 3;
```
- **没有 ICP 时**：存储引擎扫出 `a = 1` 的**所有行**，返回给 Server 层，由 Server 层判断 `c = 3`。
- **有 ICP 时**：存储引擎在扫描索引时，**直接在索引层面判断 `c = 3`**，只把满足条件的记录回表。
- **意义**：减少了回表次数，虽然扫描范围没变，但过滤动作下推到了引擎层。

### 4. 场景对比：一张表看懂各种写法的索引利用程度

| WHERE 条件 | 索引使用情况 | 扫描范围 |
| :--- | :--- | :--- |
| `a = 1 AND b = 2 AND c = 3` | ✅ 完整命中 `(a, b, c)` | 精确定位到叶子节点 |
| `a = 1 AND b = 2` | ✅ 命中 `(a, b)`，`c` 范围扫描 | 锁定 `a=1, b=2` 的连续区间 |
| `a = 1 AND c = 3` | ⚠️ **仅命中 `a`**，`c` 仅作过滤/ICP | 扫描 `a=1` 下的**所有行** |
| `b = 2 AND c = 3` | ❌ **完全不命中**（最左列 `a` 缺失） | 全索引扫描或全表扫描 |
| `a = 1 AND b > 2 AND c = 3` | ✅ 命中 `(a, b)` 范围，`c` **失效** | 扫描 `a=1` 下所有 `b>2` 的行 |

### 5. 关键记忆法则

> **最左前缀法则的“断层”效应**：
> - 索引 `(a, b, c)` 就像一段**三节棍**。
> - 第一节 `a` 必须握住，否则兵器没法用（索引失效）。
> - 握住 `a` 后，可以接着握 `b`，然后握 `c`。
> - 但如果你**跳过 `b` 直接去握 `c`**，那么你的手只能停在 `a` 的位置，`c` 那节只能当摆设（不能缩小扫描范围），最多帮你省点力气（ICP 或覆盖索引）。

### 二.8 Explain 分析：看懂 type 字段（ALL -> index -> range -> ref -> eq_ref -> const）

EXPLAIN 中的 type 字段是衡量 SQL 语句访问效率的核心指标。它反映了 MySQL 决定如何从表中查找行的方式。

从最差到最优的顺序排列如下（越往下性能越好）：

1. ALL —— 全表扫描
含义：MySQL 扫描整个表的每一行来找到匹配的记录。

场景：查询列没有索引，或者优化器认为使用索引比全表扫描更慢（比如结果集超过表数据量的 20%~30%）。

优化建议：致命缺陷，必须优化。考虑添加合适的索引。

2. index —— 全索引扫描
含义：MySQL 扫描整个索引树，读取索引的全部叶子节点。

与 ALL 的区别：ALL 读的是数据行（大 I/O），index 读的是索引叶子块（小 I/O）。常见于：

覆盖索引但缺少 WHERE 条件：SELECT age FROM user ORDER BY age（需要扫全索引排序）。
查询使用了索引列，但条件无法缩小范围（如 WHERE age > 0）。
优化建议：若数据量大，仍属于低效查询。建议增加过滤条件，将 type 提升至 range。

3. range —— 索引范围扫描
含义：基于索引的范围扫描。MySQL 先定位起点，然后顺着叶子链表向右读取直到终点。

典型操作符：>、<、BETWEEN、LIKE 'abc%'、IN (值较少时)。

场景示例：WHERE age > 20 AND age < 30（假设 age 有索引）。

优化建议：通常已能满足大部分需求，注意缩小扫描区间长度。

4. ref —— 非唯一索引等值匹配
含义：使用非唯一索引（或联合索引的最左前缀）进行 = 或 <=> 匹配。可能返回多行匹配记录。

场景示例：SELECT * FROM user WHERE name = 'Bob'（name 是非唯一索引）。

联合索引中的表现：WHERE a = 1（命中联合索引 (a, b) 的 a 部分），类型也是 ref。

5. eq_ref —— 唯一索引等值匹配（关联查询专属）
含义：在多表连接查询中，被驱动表使用主键或唯一非空索引进行等值匹配，且只返回一行数据。

场景示例：

sql
SELECT * FROM order o JOIN user u ON o.user_id = u.id;
对于 order 表中的每一行，在 user 表查 id 时，使用的是主键等值查询，保证返回 0 或 1 行。

对比 ref：eq_ref 是 ref 的特殊情况，明确知道结果最多一行（使用了唯一索引）。

6. const —— 常量匹配（主键/唯一索引等值）
含义：对主键或唯一非空索引进行常量等值比较，最多返回一行，且在查询开始时就被转换为常量。

场景示例：SELECT * FROM user WHERE id = 1。

极致优化：如果查询命中 const，这部分数据几乎不消耗额外查询时间，直接被优化器当作常数处理。

7. system（const 的特例）
含义：表中只有一行数据（系统表或极少数据的配置表），等同于访问一个常量行。

类型描述：在实际业务表里几乎见不到。

8. NULL（最佳状态）
含义：不需要访问表或索引。例如 SELECT 1 + 2 或 SELECT MAX(id) FROM user (利用索引元数据直接得出)。

总结对照表
type 字段值	核心特征	是否能有效利用索引缩小范围？	粗略性能评级
ALL	全表硬扫	❌ 无	💀 灾难级
index	全索引硬扫	❌ 无（仅避开了数据页读取）	⚠️ 较差
range	范围锁区间	✅ 部分	👍 良好
ref	非唯一键等值	✅ 高	✅ 优秀
eq_ref	联表唯一键等值	✅ 极高	🚀 卓越
const	主键常量	✅ 完美	🚀 完美

### 模块一：接口自动化（Java + RestAssured）

#### 1.1 利用 RestAssured 的 Session 机制保持登录态

**基础用法**：
```java
SessionFilter sessionFilter = new SessionFilter();

// 登录请求，携带 SessionFilter 自动捕获 Cookie
given()
    .filter(sessionFilter)
    .contentType("application/json")
    .body("{\"username\":\"admin\",\"password\":\"123456\"}")
.when()
    .post("/api/login")
.then()
    .statusCode(200);

// 后续业务请求复用同一个 SessionFilter，自动携带 Cookie
given()
    .filter(sessionFilter)
.when()
    .get("/api/user/profile")
.then()
    .statusCode(200)
    .body("data.username", equalTo("admin"));
```

**深度要点**：
- **原理**：`SessionFilter` 实现了 `Filter` 接口，底层维护 Apache HttpClient 的 `HttpContext`，通过 `CookieStore` 存储服务端返回的 `Set-Cookie`，并在后续请求中自动设置 `Cookie` 头。
- **并发注意事项**：若测试类中 `SessionFilter` 定义为静态成员，多线程并行执行时会串用同一份 Cookie，导致登录态错乱。解决方案：
  - 每个测试方法内新建 `SessionFilter` 实例。
  - 使用 `ThreadLocal` 封装 `SessionFilter`。
- **失效处理**：需配合 `Response` 的 `statusCode` 判断，若返回 401，则重新执行登录逻辑刷新 `SessionFilter`。

#### 1.2 结合 JsonPath 提取深层响应字段

**基础用法**：
```java
Response resp = given().when().get("/api/order/12345");

// 提取单值
String orderStatus = resp.jsonPath().getString("data.order.status");
int totalAmount = resp.jsonPath().getInt("data.order.totalAmount");

// 提取列表
List<String> itemNames = resp.jsonPath().getList("data.items.name");

// 条件过滤（Groovy GPath 语法）
String firstActiveItem = resp.jsonPath()
    .getString("data.items.find { it.status == 'active' }.name");
```

**深度要点**：
- RestAssured 默认使用 **Groovy 的 `JsonSlurper`**，语法与标准 Jayway JsonPath 略有不同，比如过滤条件是 `find { }` 而非 `[?(@.xxx)]`。

### 模块二：性能测试（JMeter）

#### 2.1 线程组设置与并发模型

**阶梯加压（Stepping Thread Group）**：
- 作用：模拟生产流量从低到高逐步攀升，观察系统性能拐点。
- 配置示例：
  - `This group will start 100 threads`：最终并发数 100。
  - `First, wait for 10 seconds`：准备时间。
  - `Then start 5 threads every 10 seconds`：每 10 秒增加 5 个并发。
  - 设置足够长的 `Hold load for` 使系统在峰值稳定运行一段时间。

**并发数计算（Ramp-Up + Loop Count）**：
- 普通 `Thread Group` 中：
  - 并发数 ≈ `线程数 × 循环次数 / Ramp-Up 时间`（粗略估算）
  - 若需精确控制 QPS，需配合 **Constant Throughput Timer** 或 **Throughput Shaping Timer**。

### 四.1 三次握手 & 四次挥手：各状态变化（SYN_SENT, SYN_RCVD, ESTABLISHED, FIN_WAIT_1, CLOSE_WAIT, TIME_WAIT）
TCP 连接的建立与释放是网络编程的核心基础，下面用**状态流转图**和**分步解析**来呈现三次握手与四次挥手。

### 一、三次握手 —— 建立连接

#### 1. 状态流转图

```text
Client (主动打开)                              Server (被动打开)
       |                                              |
  CLOSED                                              LISTEN
       |                                              |
   (主动发起连接)                                      |
       |                                              |
       |  ------ SYN=1, seq=x (SYN_SENT) ------->     |
       |                                              |
   SYN_SENT                                          | 
       |                                              | 收到 SYN
       |                                              | 变为 SYN_RCVD
       |                                              |
       |  <----- SYN=1, ACK=1, seq=y, ack=x+1 ------ |
       |                                              |
   (收到 SYN+ACK)                                     |
   变为 ESTABLISHED                                   |
       |                                              |
       |  ------ ACK=1, seq=x+1, ack=y+1 ------->     |
       |                                              |
       |                                          (收到 ACK)
       |                                          变为 ESTABLISHED
       |                                              |
  ESTABLISHED                                     ESTABLISHED
       |                                              |
```

#### 2. 状态与报文详解

| 阶段 | Client 状态 | Server 状态 | 发送报文 | 关键字段含义 |
| :--- | :--- | :--- | :--- | :--- |
| **第一次握手** | 发送 SYN 后进入 **`SYN_SENT`** | `LISTEN` → 收到 SYN 后进入 **`SYN_RCVD`** | Client → Server : `SYN=1, seq=x` | 同步序列号，`x` 为 Client 初始序列号 |
| **第二次握手** | `SYN_SENT` → 收到 SYN+ACK 后进入 **`ESTABLISHED`** | `SYN_RCVD` | Server → Client : `SYN=1, ACK=1, seq=y, ack=x+1` | `y` 为 Server 初始序列号，`ack=x+1` 确认收到 `x` |
| **第三次握手** | `ESTABLISHED` 发送 ACK | `SYN_RCVD` → 收到 ACK 后进入 **`ESTABLISHED`** | Client → Server : `ACK=1, seq=x+1, ack=y+1` | 确认收到 `y`，序列号同步完成 |

> **为什么是三次？**  
> 防止已失效的连接请求报文段突然又传到 Server，导致 Server 误开连接浪费资源。第三次握手让 Client 有机会确认自己确实要连接。

---

### 二、四次挥手 —— 释放连接

#### 1. 状态流转图（以 Client 主动关闭为例）

```text
Client (主动关闭)                               Server (被动关闭)
       |                                              |
  ESTABLISHED                                    ESTABLISHED
       |                                              |
   (调用 close)                                       |
       |                                              |
  FIN_WAIT_1                                         |
       |                                              |
       |  ------ FIN=1, seq=u --------------->       |
       |                                              |
       |                                          (收到 FIN)
       |                                          变为 CLOSE_WAIT
       |                                          (应用层可能还在发数据)
       |                                              |
       |  <------ ACK=1, seq=v, ack=u+1 --------     |
       |                                              |
   (收到 ACK)                                         |
   变为 FIN_WAIT_2                                    |
       |                                              |
       |                                          (应用层调用 close)
       |                                          变为 LAST_ACK
       |                                              |
       |  <------ FIN=1, ACK=1, seq=w, ack=u+1 --    |
       |                                              |
   (收到 FIN)                                         |
   变为 TIME_WAIT                                     |
   等待 2MSL 后                                       |
   变为 CLOSED                                        |
       |                                          (收到 ACK)
       |                                          变为 CLOSED
       |                                              |
```

#### 2. 状态与报文详解

| 阶段 | Client 状态 | Server 状态 | 发送报文 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| **第一次挥手** | 发送 FIN 后进入 **`FIN_WAIT_1`** | `ESTABLISHED` → 收到 FIN 后进入 **`CLOSE_WAIT`** | Client → Server : `FIN=1, seq=u` | Client 告知自己已无数据要发 |
| **第二次挥手** | `FIN_WAIT_1` → 收到 ACK 后进入 **`FIN_WAIT_2`** | `CLOSE_WAIT` | Server → Client : `ACK=1, seq=v, ack=u+1` | Server 确认收到 FIN，但可能还有数据要发给 Client |
| **第三次挥手** | `FIN_WAIT_2` | `CLOSE_WAIT` → 应用层也调用 close 后发送 FIN，进入 **`LAST_ACK`** | Server → Client : `FIN=1, ACK=1, seq=w, ack=u+1` | Server 也发 FIN 告知无数据要发 |
| **第四次挥手** | `FIN_WAIT_2` → 收到 FIN 后进入 **`TIME_WAIT`** | `LAST_ACK` → 收到 ACK 后进入 `CLOSED` | Client → Server : `ACK=1, seq=u+1, ack=w+1` | Client 确认收到 FIN，进入 `TIME_WAIT` 等待 2MSL 后关闭 |

> **为什么是四次？**  
> TCP 是全双工通信，一方发送 FIN 只表示自己不再发数据，但仍可接收数据。因此另一方的 ACK 和 FIN 通常分开发送（中间可能还有数据传输），所以需要四次交互。

> **`TIME_WAIT` 为什么是 2MSL？**  
> - **确保最后一个 ACK 能到达 Server**：若 ACK 丢失，Server 会在 1MSL 内重传 FIN，Client 可以重发 ACK。  
> - **让旧连接的报文从网络中消失**：2MSL 时间后，网络中不会再有该连接残留的报文，避免干扰后续新连接。

### 三、状态速记口诀

```
三次握手：客户发 SYN 进 SENT，服务收 SYN 进 RCVD，服务回 SYN+ACK 仍 RCVD，
          客户收 SYN+ACK 进 ESTAB，客户发 ACK 亦 ESTAB，服务收 ACK 终 ESTAB。

四次挥手：主动发 FIN 进 FIN1，被动收 FIN 进 CLOSE，被动回 ACK 仍 CLOSE，
          主动收 ACK 进 FIN2，被动再发 FIN 进 LAST，主动收 FIN 进 TIME，
          等待 2MSL 变 CLOSED，被动收 ACK 直接 CLOSED。
```
### 四.2 TIME_WAIT 过多的危害：占用端口资源，解决办法（开启 tcp_tw_reuse）
在 TCP 连接的主动关闭方，**`TIME_WAIT`** 状态是确保连接可靠终止的必要设计。但在高并发短连接场景（如 Nginx 反向代理、爬虫频繁请求、压力测试客户端）下，如果出现大量 `TIME_WAIT` 堆积，会引发严重的资源问题。

### 1. `TIME_WAIT` 过多的具体危害

#### 危害一：耗尽本地端口资源（最致命）
- **原理**：TCP 连接由四元组唯一标识：`(源IP, 源端口, 目的IP, 目的端口)`。
- **冲突条件**：对于同一个目标 IP 和端口，如果本地使用固定的源 IP（如只有一块网卡），则**可用的源端口范围受限**（默认 `net.ipv4.ip_local_port_range = 32768 60999`，约 2.8 万个）。
- **现象**：当 `TIME_WAIT` 连接占满所有可用端口后，新的主动连接会返回 `Cannot assign requested address` 错误，服务直接不可用。

#### 危害二：内存与连接跟踪表压力
- 每个 `TIME_WAIT` 状态的 socket 虽然不传输数据，但内核仍为其保留一个 **`tcp_timewait_sock`** 结构体，占用约 200~300 字节内存。
- 在启用 Netfilter 或连接跟踪 (nf_conntrack) 的 Linux 系统上，`TIME_WAIT` 会挤占 `nf_conntrack` 表项，可能导致 `nf_conntrack: table full, dropping packet` 日志出现，进而影响 NAT 转发或防火墙规则。

#### 危害三：端口复用冲突
- 若应用层强行 `bind` 复用处于 `TIME_WAIT` 的端口（未设置 `SO_REUSEADDR`），会触发 `Address already in use` 异常。
- 即使设置 `SO_REUSEADDR`，也仅限于**重启监听端口**的场景，对于**主动连接**的源端口复用，必须依赖 `tcp_tw_reuse` 或 `tcp_tw_recycle`（已废弃）。

---

### 2. 解决办法详解

#### 方案一：开启 `net.ipv4.tcp_tw_reuse`（最直接有效）

**作用机制**：
- 允许内核在**客户端主动发起连接**时，复用处于 `TIME_WAIT` 状态的**源端口**去连接**不同的目标 IP 或端口**。
- **核心前提**：必须同时启用 **TCP 时间戳选项** (`net.ipv4.tcp_timestamps = 1`)，否则该参数无效。

**配置命令**：
```bash
echo 1 > /proc/sys/net/ipv4/tcp_tw_reuse
# 持久化写入 /etc/sysctl.conf:
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_timestamps = 1
```

**关键限制（面试必问）**：
- **只对主动连接方（客户端）生效**，服务器端监听的端口无法通过该参数减少 `TIME_WAIT`。
- **仅当 `TIME_WAIT` 连接的初始时间戳严格小于当前时间戳时才能复用**，避免了旧报文误入新连接的潜在风险。
- **不能复用完全相同的四元组去连接同一个目标**（即 `(src_ip, src_port, dst_ip, dst_port)` 完全相同仍会失败），因为 TCP 时间戳机制只能区分不同时间建立的连接，若目标相同且端口相同，序列号仍有混淆风险。

#### 方案二：调整系统内核参数组合拳

| 参数 | 推荐值 | 作用 |
| :--- | :--- | :--- |
| `net.ipv4.tcp_tw_reuse` | `1` | 客户端复用 `TIME_WAIT` 端口（需配合时间戳） |
| `net.ipv4.tcp_timestamps` | `1` | 启用 TCP 时间戳，`tw_reuse` 的前提 |
| `net.ipv4.tcp_fin_timeout` | `30` | 缩短 `FIN_WAIT_2` 状态的超时时间（从默认 60s 降到 30s），间接减少 `TIME_WAIT` 前序状态的滞留 |
| `net.ipv4.ip_local_port_range` | `1024 65535` | 扩大可用临时端口范围 |
| `net.ipv4.tcp_max_tw_buckets` | `262144` | 限制系统允许的 `TIME_WAIT` 最大数量，超出的直接销毁（治标不治本，紧急兜底） |

**⚠️ 注意**：**`tcp_tw_recycle` 已在 Linux 4.12 内核中被彻底移除**，因其在 NAT 环境下会导致时间戳错乱丢包，面试时主动提及该知识点的演进会是加分项。

#### 方案三：应用层优化——变短连接为长连接

- **HTTP/1.1 `Connection: keep-alive`**：复用已建立的 TCP 连接发送多次请求，从源头避免频繁创建连接。
- **HTTP/2 多路复用**：一个 TCP 连接上并发传输多个流，进一步减少连接数。
- **连接池技术**：数据库连接池（HikariCP）、HTTP 客户端连接池（Apache HttpClient PoolingHttpClientConnectionManager），均是通过复用连接减少 `TIME_WAIT` 产生。

#### 方案四：调整内核的孤儿连接回收策略（仅限服务端被动关闭）

如果问题是**服务器端**因主动关闭连接而产生大量 `TIME_WAIT`（比如 Nginx 反向代理后端），可以：
- 配置 **`net.ipv4.tcp_max_orphans`** 限制孤儿 socket 数量。
- **让客户端主动关闭连接**：Nginx 配置 `proxy_ignore_client_abort on` 或调整应用逻辑，让发起请求的一方（如浏览器、APP）成为主动关闭方，将 `TIME_WAIT` 分散到海量客户端上。

---

### 3. 总结：一句话面试回答模板

> **问**：`TIME_WAIT` 过多怎么办？  
> **答**：  
> “主要危害是耗尽客户端可用端口导致 `Cannot assign requested address`。解决办法分三层：
> 1. **内核参数**：开启 `tcp_tw_reuse` 配合 `tcp_timestamps` 允许客户端复用端口。
> 2. **架构调整**：使用长连接（Keep-Alive）或连接池从源头减少连接创建。
> 3. **策略分流**：如果是服务器端问题，尽量让客户端成为主动关闭方，将 `TIME_WAIT` 分散到客户端侧。”

### 四.3 流量控制：滑动窗口机制，接收方窗口 rwnd 为 0 时发送方的坚持定时器
TCP 的**流量控制**与拥塞控制不同，它的目的纯粹是**防止发送方发的太快，导致接收方缓冲区溢出而丢包**。其核心实现依赖于**滑动窗口机制**，并由**坚持定时器**解决零窗口死锁问题。

### 一、滑动窗口机制

#### 1. 基本概念
- **接收窗口 (rwnd, Receive Window)**：接收方在 TCP 报文头部的“窗口字段”中告诉发送方，自己**接收缓冲区还剩多少可用空间**。
- **发送窗口 (Send Window)**：发送方根据 `rwnd` 动态调整自己的发送上限，保证在途未确认的数据量 ≤ `rwnd`。

#### 2. 工作流程示意图
```text
发送方                                    接收方
   |                                        |
   | <---- ACK, win=4096 (rwnd=4096) ------ |  接收方通告窗口
   |                                        |
   | 发送 1-1024 (seq=1, len=1024)          |
   | 发送 1025-2048                         |
   | 发送 2049-3072                         |
   | 发送 3073-4096                         |  (发送窗口已满，暂停发送)
   |                                        |
   | <---- ACK=2049, win=2048 ------------- |  应用读取了前2048字节，窗口向右滑动
   |                                        |
   | 发送 4097-5120                         |  继续发送
   |                                        |
```

- **窗口滑动**：每当接收方确认了部分数据，窗口就会向右“滑动”，腾出空间让发送方发送新数据。
- **动态调整**：`rwnd` 随接收方应用读取速度实时变化，发送方**严格服从**，绝不超发。

### 二、零窗口与坚持定时器

#### 1. 零窗口死锁问题
当接收方缓冲区满，`rwnd = 0` 时：
1. 发送方收到 `win=0` 的 ACK，停止发送数据。
2. 接收方应用层读取数据，缓冲区腾出空间，**发送一个 `win > 0` 的 ACK 通告新窗口**。
3. **但这个 ACK 在传输中丢失了！**
4. 发送方一直等待非零窗口通告，接收方一直等待发送方发数据，**双方陷入死锁**。

#### 2. 坚持定时器的原理与作用
为防止上述死锁，TCP 为发送方引入**坚持定时器**：
- **触发条件**：当发送方收到 **`rwnd = 0`** 的 ACK 时，启动坚持定时器。
- **定时器动作**：超时后，发送方会主动发送一个**零窗口探测报文**（通常为 1 字节数据或纯 ACK 探测）。
- **接收方响应**：接收方收到探测报文后，必须回复当前的最新窗口值 `rwnd`。
- **持续探测**：若回复的 `rwnd` 仍为 0，则重置定时器（通常按指数退避增加间隔），继续周期性探测，直到窗口打开。

#### 3. 零窗口探测过程的时序图
```text
发送方                                    接收方
   |                                        |
   | <------- ACK, win=0 ------------------ |  缓冲区满，rwnd=0
   |                                        |
   | [启动坚持定时器]                        |
   |                                        |  应用读取数据，rwnd=4096
   |                                        |  发送 ACK, win=4096  ----> (丢失!)
   |                                        |
   | [坚持定时器超时]                        |
   | ------- 探测报文 (1字节) ------------>  |
   |                                        |
   | <------ ACK, win=4096 ---------------- |  窗口打开，恢复正常传输
   |                                        |
   | 发送数据 ...                            |
```

### 三、与拥塞控制的区别（面试高频对比）

| 对比项 | 流量控制 | 拥塞控制 |
| :--- | :--- | :--- |
| **目的** | 匹配**接收方**的处理能力 | 适应**网络链路**的承载能力 |
| **核心变量** | `rwnd` (接收方窗口) | `cwnd` (拥塞窗口) |
| **信息来源** | 接收方通告的窗口字段 | 由发送方根据丢包、RTT 等推测 |
| **发送上限** | `min(rwnd, cwnd)` |  |
| **零窗口处理** | 坚持定时器探测 | 不适用 |

### 四、关键追问准备

**问：坚持定时器探测时，为什么通常发 1 字节数据？**
- 纯 ACK 报文在传输中丢失后，接收方**不会重传 ACK**（ACK 不消耗序列号，不保证可靠送达）。
- 携带 1 字节数据的报文有序列号，若丢失会触发超时重传，确保探测能被接收方处理并回复最新窗口。即使数据本身无用，接收方也会丢弃重复数据但照常回复 ACK。

**问：如果接收方回复的 `rwnd` 依然为 0，坚持定时器怎么调整？**
- 采用**指数退避**，如首次超时 5 秒，第二次 10 秒，第三次 20 秒……上限通常为 60 秒。既保证死锁能被解除，又避免过于频繁的探测浪费网络资源。

### 四.4 拥塞控制：慢启动、拥塞避免、快重传、快恢复
TCP 的**拥塞控制**由四个核心算法构成，共同目标是**在不引发网络崩溃的前提下，最大化利用带宽**。这四个算法并非孤立存在，而是根据网络状况动态切换的一套状态机。

### 一、核心变量与状态转换总览

| 变量 / 状态 | 含义 |
| :--- | :--- |
| **`cwnd`** (拥塞窗口) | 发送方自己维护的拥塞窗口，限制**已发送但未确认的数据量**上限。 |
| **`ssthresh`** (慢启动阈值) | 区分“慢启动”和“拥塞避免”的分水岭。 |
| **`rwnd`** | 接收方通告窗口，最终发送窗口 `= min(cwnd, rwnd)`。 |

**算法切换总览**：

```text
[ 连接建立 ] → 慢启动 (cwnd < ssthresh)
                    │
                    │ (cwnd 达到 ssthresh 或 发生丢包)
                    ▼
              拥塞避免 (cwnd >= ssthresh)
                    │
        ┌───────────┴───────────┐
        │ (收到 3 个重复 ACK)    │ (RTO 超时)
        ▼                       ▼
     快重传                    慢启动 (cwnd=1, ssthresh 减半)
        │
        ▼
     快恢复 (cwnd 减半后线性增长)
```

---

### 二、四个算法逐一详解

#### 1. 慢启动

**触发时机**：
- TCP 连接刚建立时。
- 发生 **RTO 超时** 后重新开始传输。

**工作过程**：
- 初始 `cwnd` 通常为 **1 个 MSS** (最大报文段长度，实际 Linux 初始为 10)。
- 每收到一个 **新的 ACK**，`cwnd` **翻倍**（指数增长）。
- 增长曲线：`1 → 2 → 4 → 8 → 16 ...`

**退出条件**：
- `cwnd` 达到 **`ssthresh`** 时，转入**拥塞避免**阶段。
- 发生**丢包**（超时或收到 3 个重复 ACK）时，触发相应恢复机制。

```text
   cwnd
    │
    │                     *  (达到 ssthresh，切换)
    │                   *
    │                 *
    │               *
    │             *
    │           *
    │         *
    │       *
    │     *
    │   *
    │ *
    └─────────────────────────── 时间
          (指数增长)
```

#### 2. 拥塞避免

**触发时机**：
- `cwnd >= ssthresh` 时自动进入。

**工作过程**：
- 每经过一个完整的 **RTT**（即收到该窗口内所有数据的 ACK），`cwnd` **增加 1 个 MSS**（线性增长）。
- 实际实现中，每个 ACK 收到时 `cwnd += (MSS * MSS / cwnd)`，近似每个 RTT 加 1。

**退出条件**：
- 若收到 **3 个重复 ACK**（表示中间有报文丢失，但后续报文到达了），`ssthresh` 减半，转入**快恢复**（Reno 版本）。
- 若发生 **RTO 超时**（严重拥塞），`ssthresh` 设为当前 `cwnd` 的一半，`cwnd` 重置为 1，退回到**慢启动**。

#### 3. 快重传

**触发条件**：发送方**连续收到 3 个相同的 ACK**（即第四个重复 ACK 到来时）。

**原理**：
- 重复 ACK 意味着接收方收到了失序的报文，说明**中间某个报文丢失了，但后续报文已到达**，网络并未完全瘫痪。
- 发送方**不必等待 RTO 超时**，立即重传那个缺失的报文段。

**示例**：
```text
发送方连续发送：seq=1, 1001, 2001, 3001, 4001
假设 seq=1001 丢失。
接收方收到 2001 后，回复 ACK=1001 (重复1)
收到 3001 后，回复 ACK=1001 (重复2)
收到 4001 后，回复 ACK=1001 (重复3)  → 触发快重传，立即重传 seq=1001
```

#### 4. 快恢复

**触发条件**：快重传完成后立即进入（Reno 版本）。

**工作过程**：
1. 将 **`ssthresh`** 设置为当前 `cwnd` 的一半。
2. 将 **`cwnd`** 设置为 `ssthresh + 3 * MSS`（补偿已经离开网络的重复 ACK 数量）。
3. 进入**拥塞避免**阶段，后续每收到一个重复 ACK，`cwnd` 临时加 1；收到新 ACK 后，`cwnd` 回落至 `ssthresh`，正式开始线性增长。

**与超时恢复的本质区别**：
| 对比项 | 快重传 + 快恢复 | RTO 超时 |
| :--- | :--- | :--- |
| **触发** | 收到 3 个重复 ACK | 重传计时器超时 |
| **网络判断** | 轻度拥塞（部分丢包） | 重度拥塞（网络几乎瘫痪） |
| **`cwnd` 变化** | 减半后继续线性增长 | 直接降到 1，重新慢启动 |
| **性能影响** | 较高，避免盲目降速 | 较低，丢包后恢复慢 |

---

### 三、完整状态流转实例

假设初始 `cwnd = 1 MSS`，`ssthresh = 16 MSS`，采用 Reno 算法：

1. **慢启动**：`cwnd` 指数增长 `1 → 2 → 4 → 8 → 16`。
2. 到达 `ssthresh=16`，进入**拥塞避免**，`cwnd` 线性增长 `17 → 18 → 19 ...`
3. 当 `cwnd = 20` 时，收到 **3 个重复 ACK**：
   - `ssthresh = 20 / 2 = 10`
   - `cwnd = 10 + 3 = 13`，进入**快恢复**
4. 收到新 ACK 后，`cwnd = 10`，转入**拥塞避免**，继续线性增长。

若在第 3 步发生的是 **RTO 超时**而非重复 ACK：
   - `ssthresh = 20 / 2 = 10`
   - `cwnd = 1`，回到**慢启动**重新开始。

---

### 四、Tahoe 与 Reno 的演进区别

| 算法版本 | 收到 3 个重复 ACK 后的行为 | 优缺点 |
| :--- | :--- | :--- |
| **Tahoe** | 无论超时还是重复 ACK，一律 `cwnd=1` 慢启动 | 简单但过于保守，轻微丢包也导致吞吐量骤降 |
| **Reno** | 重复 ACK 触发**快重传+快恢复**，`cwnd` 减半后线性增长 | 对轻度拥塞更友好，是现代 TCP 的基础 |

### 五、面试常见追问

**问：为什么重复 ACK 要等到 3 个才触发快重传？**
- 1~2 个重复 ACK 可能由**报文乱序**引起，并非真正丢包。
- 3 个重复 ACK 几乎能确定是**丢包**（因为后续报文已经到达 3 个），此时立即重传可避免 RTO 长时间等待。

**问：快恢复时 `cwnd = ssthresh + 3` 的“+3”是什么含义？**
- 因为已经收到了 3 个重复 ACK，意味着有 3 个报文段**离开了网络**（被接收方缓存），网络腾出了 3 个 MSS 的容量，因此 `cwnd` 可临时增加 3，允许继续发送少量新数据而不违反拥塞控制原则。

**问：Google BBR 与传统的 Reno/CUBIC 有什么不同？**
- 传统拥塞控制**基于丢包**作为拥塞信号，BBR **基于带宽和 RTT 探测**，在有一定丢包率的网络（如无线网）中吞吐量远优于传统算法。

### 四.5 粘包/拆包：原因（TCP 流式协议），解决办法（固定长度、分隔符、Header+Body 长度字段）
TCP 是**面向字节流**的协议，数据在发送方和接收方之间如同自来水一样**没有边界**。应用层写入 100 字节，接收方读取时可能分两次收到（拆包），也可能和下一条消息合并收到（粘包）。理解其本质并掌握解决方案是网络编程的基本功。

### 一、粘包/拆包的根本原因

| 层面 | 原因 |
| :--- | :--- |
| **发送方（应用层）** | Nagle 算法会合并多个小包为一个 MSS 发送。 |
| **接收方（应用层）** | `read(buffer)` 返回的是内核 Socket 缓冲区当前可用的数据量，与应用层消息边界无关。 |
| **TCP 本身** | TCP 是流式协议，只保证字节流的顺序和可靠送达，**不保留消息边界**。 |

**通俗比喻**：寄快递时每个包裹有独立的快递单和包装（UDP 面向报文），而 TCP 是**用一根水管传输连续的水流**，接水的人需要自己判断什么时候是一杯水满了。

### 二、三种经典解决方案

#### 1. 固定长度（FixedLengthFrameDecoder）

**原理**：约定每条消息的字节数完全相同，不够的用空格或 `0` 填充。接收方每次读取固定长度字节作为一个完整消息。

**适用场景**：消息长度恒定的协议，如某些传感器数据、简单的控制指令。

**示例**：约定每条消息固定 10 字节。
```
发送： "HELLO" + 5个空格
       "WORLD" + 5个空格
接收： 每次读10字节，分别解析为 "HELLO" 和 "WORLD"
```

**Netty 实现**：
```java
pipeline.addLast(new FixedLengthFrameDecoder(10));
```

**优点**：解析极简，性能极高。  
**缺点**：浪费带宽，长度固定导致灵活性差。

---

#### 2. 分隔符（DelimiterBasedFrameDecoder）

**原理**：在每个消息末尾添加一个**特殊的分隔符**（如 `\r\n`、`\0`），接收方扫描缓冲区，一旦遇到分隔符即提取前面的数据作为完整消息。

**适用场景**：文本协议（如 Redis、SMTP、HTTP/1.x 的行分隔）、简单的自定义命令协议。

**示例**：以 `\n` 分隔。
```
发送： "LOGIN admin 123456\n"
       "GET_USER_INFO\n"
接收： 扫描到第一个 `\n`，提取 "LOGIN admin 123456"；
       扫描到第二个 `\n`，提取 "GET_USER_INFO"。
```

**Netty 实现**：
```java
// 以 \r\n 分隔，最大长度 8192
pipeline.addLast(new DelimiterBasedFrameDecoder(8192, Delimiters.lineDelimiter()));
// 自定义分隔符
pipeline.addLast(new DelimiterBasedFrameDecoder(8192, Unpooled.copiedBuffer("|".getBytes())));
```

**注意**：若消息内容中可能包含分隔符，需进行**转义**（如用 `\n` 替换为 `\\n`），否则会被误判为边界。

---

#### 3. 长度字段（LengthFieldBasedFrameDecoder）—— **最通用、最推荐**

**原理**：在消息头部添加一个**固定长度的长度字段**，标识整个消息体（或头部+消息体）的字节数。接收方先读取长度字段，再根据该长度读取后续的完整消息体。

**格式**：
```
+--------+------------------+
| Length |   Message Body   |
+--------+------------------+
   2字节       N 字节
```

**适用场景**：绝大多数自定义二进制协议（如 Dubbo、gRPC、自定义 RPC）。

**Netty 实现**（核心参数解析）：
```java
pipeline.addLast(new LengthFieldBasedFrameDecoder(
    65535,                  // maxFrameLength：最大包长，防止内存溢出
    0,                      // lengthFieldOffset：长度字段在报文中的偏移量
    4,                      // lengthFieldLength：长度字段自身占用字节数（常用 int 即 4 字节）
    0,                      // lengthAdjustment：长度调整值（若长度字段包含头部自身长度则需调整）
    4                       // initialBytesToStrip：解码后丢弃头部长度字段，只留消息体
));
```

**常见变体**：
- 长度字段包含头部：`LengthAdjustment = - (lengthFieldLength)`，调整后读取正确的消息体。
- 头部有多个字段（如 Magic、版本号），只需指定正确的 `lengthFieldOffset`。

**优点**：灵活、节省带宽、支持任意长度的消息。  
**缺点**：实现比前两者稍复杂。

---

### 三、方案对比与选型指南

| 方案 | 带宽效率 | 实现复杂度 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **固定长度** | 低（需填充） | 极简 | 定长指令、传感器数据 |
| **分隔符** | 中（分隔符占用） | 简单 | 文本协议、控制命令 |

### 四、结合简历场景延伸（智能体问答流式输出）

你的简历中提到了 **WebSocket 流式输出** 和 **智能体问答**。在该场景下：

- **WebSocket 本身就是基于 TCP 的流式协议**，同样存在粘包问题。
- 浏览器端的 WebSocket API 底层已经处理了消息边界（WebSocket 协议帧中有 `payload length` 字段），所以前端 `onmessage` 每次收到的都是完整的一帧消息。
- 但如果你在后端用 Netty 自己实现 WebSocket 协议，就需要用 **长度字段方案** 解析 WebSocket 帧格式（帧头包含 `payload len` 字段）。
- 若你用的是 SSE（Server-Sent Events），则天然基于**分隔符**（两个连续的 `\n\n` 分隔每条事件）。

**面试回答话术**：
> “在实现自研 RPC 框架或网络通信组件时，我使用 Netty 的 `LengthFieldBasedFrameDecoder` 处理粘包，通过定义 `length` 字段标识包体长度，既节省带宽又能适应动态消息大小。在 WebSocket 场景下，协议层已包含长度字段，因此应用层无需额外处理。”

### 四.6 HTTP/1.1 优化：长连接 (Connection: keep-alive)、管线化 (Pipelining 已废弃)、HOST 头

HTTP/1.1 相较于 HTTP/1.0 做了三项关键优化，显著提升了 Web 性能和虚拟主机的支持能力。下面逐一拆解其原理、用法与局限性。

### 一、长连接：`Connection: keep-alive`

#### 1. 背景：HTTP/1.0 的短连接之痛
HTTP/1.0 默认**每请求一次资源就新建一个 TCP 连接**，传输完毕后立即关闭。一个包含 10 张图片的网页需要建立 11 次 TCP 连接（1 次 HTML + 10 次图片），带来巨大开销：
- 重复三次握手和四次挥手的网络耗时。
- 大量 `TIME_WAIT` 状态占用端口资源。

#### 2. HTTP/1.1 的长连接机制
HTTP/1.1 默认启用 **持久连接**（Persistent Connection），响应头中通常包含：
```
Connection: keep-alive
```
- **行为**：完成一次请求-响应后，TCP 连接**不会立即关闭**，而是保持打开状态，供后续 HTTP 请求复用。
- **优势**：
  - 减少 TCP 握手/挥手次数，降低延迟。
  - 利用 TCP 已建立的拥塞窗口，避免慢启动惩罚。
- **关闭条件**：
  - 客户端或服务端发送 `Connection: close` 头部明确断开。
  - 空闲超时（服务端通常配置 `KeepAliveTimeout`，如 Nginx 默认 75 秒）。
  - 达到最大请求数限制（如 Apache 的 `KeepAliveRequests`）。

#### 3. 局限性
长连接仅解决了**连接复用**问题，但请求-响应仍是**串行**的：前一个请求的响应返回后，才能在同一连接上发送下一个请求（队头阻塞）。

---

### 二、管线化：Pipelining（已废弃）

#### 1. 原理
HTTP/1.1 定义了 **管线化** 技术，允许客户端在一个持久连接上**连续发送多个请求而不等待响应**，服务端必须按照请求的发送顺序依次返回响应。

```text
[传统长连接]
Client --- Req1 --->
       <--- Resp1 ---
                     --- Req2 --->
                     <--- Resp2 ---

[管线化]
Client --- Req1 --->--- Req2 --->
       <--- Resp1 ---<--- Resp2 ---
```

#### 2. 为何被废弃？
| 问题 | 具体描述 |
| :--- | :--- |
| **队头阻塞 (HOL Blocking)** | 若第一个响应处理慢（如查询大文件），后续响应即使已准备好也必须排队等待，导致整体延迟增加。 |
| **代理/服务器支持差** | 多数中间代理无法正确处理管线化请求，可能导致响应错乱或连接断开。 |
| **浏览器默认关闭** | Chrome、Firefox 等主流浏览器均**禁用**或**移除**了管线化支持。 |
| **幂等性风险** | 非幂等请求（如 POST）若因连接断开而重传，可能引发副作用。 |

#### 3. 现状
管线化已被 **HTTP/2 的多路复用** 彻底取代，后者通过**二进制分帧**和**流优先级**真正解决了队头阻塞问题。

---

### 三、`Host` 头：虚拟主机的基石

#### 1. 问题背景
在 HTTP/1.0 时代，一个 IP 地址只能对应一个域名。若想在同一台服务器上托管多个网站（如 `a.com` 和 `b.com`），必须为每个网站分配独立 IP，造成 IP 地址严重浪费。

#### 2. HTTP/1.1 的 `Host` 头机制
HTTP/1.1 **强制要求**请求头中必须包含 `Host` 字段，指明目标域名。

```http
GET /index.html HTTP/1.1
Host: www.example.com
```

#### 3. 工作原理
1. 客户端发起 HTTP 请求时，在 `Host` 头中携带要访问的域名。
2. 服务端接收请求后，解析 `Host` 头，根据域名将请求**路由到对应的虚拟主机（Virtual Host）** 目录。
3. 即使 `a.com` 和 `b.com` 解析到同一 IP，服务器也能返回正确的网站内容。

**示例**：Nginx 配置两个虚拟主机
```nginx
server {
    listen 80;
    server_name a.com;
    root /var/www/a;
}
server {
    listen 80;
    server_name b.com;
    root /var/www/b;
}
```

#### 4. 重要性
- **节省 IPv4 地址**：一个 IP 托管无数个网站。
- **CDN 与反向代理**：`Host` 头是反向代理将请求转发到正确后端源站的关键依据。

---

### 四、补充：HTTP/1.1 其他重要优化

| 特性 | 说明 |
| :--- | :--- |
| **分块传输编码 (Chunked Transfer Encoding)** | 允许服务端在不知道响应总长度时**边生成边发送**（如流式输出）。响应头包含 `Transfer-Encoding: chunked`。 |
| **缓存控制 (Cache-Control)** | 引入更精细的缓存策略头，取代 HTTP/1.0 的 `Expires`。 |
| **范围请求 (Range Request)** | 支持断点续传，请求头 `Range: bytes=0-499`。 |

### 五、面试回答话术

> **问**：HTTP/1.1 相比 HTTP/1.0 有哪些关键优化？  
> **答**：
> “主要有三点：
> 1. **默认长连接**：通过 `Connection: keep-alive` 复用 TCP 连接，减少握手开销。
> 2. **强制 `Host` 头**：使得一台物理服务器可以承载多个虚拟主机，节省 IP 地址。
> 3. **管线化（已废弃）**：虽然支持连续发送请求，但因队头阻塞问题被 HTTP/2 多路复用替代。
> 此外还增加了分块传输、更精细的缓存控制等特性。”

### 四.7 HTTPS 加密过程：非对称加密交换对称密钥 -> 对称加密传输数据
HTTPS 并非单一的加密协议，而是 **HTTP over TLS/SSL**。它的核心设计理念是：**用非对称加密安全地协商出一把临时的对称密钥，随后用这把对称密钥高效地加密业务数据**。

### 1. 宏观流程：两阶段加密

```text
[ 客户端 ]                                 [ 服务器 ]
     |                                          |
     | ① 非对称加密阶段（身份验证 + 密钥交换）     |
     | <-------------------------------------->  |
     |                                          |
     | ② 对称加密阶段（应用数据传输）             |
     | <======================================>  |
     |                                          |
```

### 2. 第一阶段：非对称加密交换对称密钥（TLS 握手核心）

此阶段的目标是**验证服务器身份**，并让双方安全地获得**同一把对称密钥**。

#### 步骤 1：客户端 Client Hello
- 发送支持的 TLS 版本、加密套件列表（如 `TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256`）、随机数 `Random_C`。

#### 步骤 2：服务端 Server Hello
- 返回确认的 TLS 版本、选中的加密套件、随机数 `Random_S`。

#### 步骤 3：服务端发送证书
- 服务器发送 **数字证书**（包含公钥、域名、CA 签名）。
- **客户端验证证书链**：用浏览器/操作系统内置的 CA 根证书验证服务器证书的签名有效性、是否过期、域名是否匹配。

#### 步骤 4：密钥交换（核心差异：RSA vs ECDHE）

这是**非对称加密真正介入**的环节，主流有两种方式：

| 方式 | 过程 | 前向安全性 | 现状 |
| :--- | :--- | :---: | :--- |
| **RSA 密钥交换** | 客户端生成**预主密钥 (Pre-Master Secret)**，用证书中的 **RSA 公钥加密**传给服务端，服务端用私钥解密。 | ❌ **无**：一旦私钥泄露，历史会话均可解密。 | 已被逐渐淘汰 |
| **ECDHE 密钥交换** | 双方通过椭圆曲线算法**临时生成公私钥对**，在不传输私钥的情况下协商出相同的**预主密钥**。 | ✅ **有**：每次会话密钥独立，私钥泄露不影响历史数据。 | **现代主流** |

#### 步骤 5：生成对称密钥
双方根据 `Random_C`、`Random_S` 和 **预主密钥**，使用 **PRF (伪随机函数)** 计算出最终的 **会话密钥 (Session Key)**。此后，非对称加密任务完成，公私钥退场。

### 3. 第二阶段：对称加密传输数据

- **算法**：通常采用 **AES-GCM** 或 **ChaCha20-Poly1305**。
- **优势**：对称加密（如 AES）的加解密速度比非对称加密（如 RSA）快 **1000 倍以上**，适合大规模数据传输。
- **完整性保护**：对称密钥同时派生出 MAC 密钥，使用 **AEAD 模式** 确保数据在传输中未被篡改。

### 4. 完整握手时序图（以 ECDHE 为例）

```text
Client                                                  Server
   |                                                      |
   |  ClientHello (Random_C, 支持的套件)                  |
   | ---------------------------------------------------> |
   |                                                      |
   |  ServerHello (Random_S, 选中套件)                    |
   |  Certificate (包含公钥)                              |
   |  ServerKeyExchange (ECDHE 临时公钥)                  |
   |  ServerHelloDone                                      |
   | <--------------------------------------------------- |
   |                                                      |
   |  ClientKeyExchange (ECDHE 临时公钥)                  |
   |  [双方各自计算 Pre-Master]                           |
   |  [双方各自计算 Session Key]                          |
   |  ChangeCipherSpec (告知后续加密)                     |
   |  Finished (握手校验)                                 |
   | ---------------------------------------------------> |
   |                                                      |
   |  ChangeCipherSpec, Finished                          |
   | <--------------------------------------------------- |
   |                                                      |
   |  ######## 对称加密传输应用数据 ########               |
   | <==================================================> |
```

### 5. 面试追问预备

**问：为什么不全用非对称加密，非要换成对称加密？**
- **性能**：RSA 2048 加解密消耗 CPU 资源巨大，且数据膨胀（RSA 密文长度等于密钥长度）。对称加密通过 CPU 硬件指令（AES-NI）加速，速度极快。

**问：抓包工具（如 Charles）为什么能解密 HTTPS？**
- 客户端**主动信任**了 Charles 的自签名根证书。Charles 作为中间人对客户端伪装成服务器，对服务器伪装成客户端，分别建立两段独立的 TLS 连接。**这是一种主动授信行为，而非协议漏洞**。

**问：SSL/TLS 握手消耗几个 RTT？**
- **TLS 1.2 完整握手**：2 个 RTT（ClientHello → ServerHello... 1 RTT；ClientKeyExchange... Finished 1 RTT）。
- **TLS 1.3 优化**：将 ClientKeyExchange 提前，握手压缩至 **1 RTT**，且支持 **0-RTT 重连**。

### 四.8 重点掌握 CA 证书链验证逻辑

CA 证书链验证是 HTTPS 安全性的基石，它解决了一个根本问题：**客户端凭什么信任一个从未谋面的服务器公钥？** 答案是通过一条**由受信任的根证书颁发机构（Root CA）背书的签名链**。

### 一、核心组件：数字证书包含什么？

一张标准的 X.509 数字证书包含以下关键字段：

| 字段 | 作用 |
| :--- | :--- |
| **Subject（主体）** | 证书持有者的身份信息（如 `CN=www.example.com`）。 |
| **Issuer（颁发者）** | 签发此证书的上级 CA 的身份信息。 |
| **Public Key（公钥）** | 主体的公钥（服务器用对应私钥证明身份）。 |
| **Signature（数字签名）** | **Issuer** 用自己的私钥对证书内容的摘要进行加密生成的值。 |
| **Validity（有效期）** | 证书的生效和失效时间。 |
| **Extensions（扩展项）** | 包含 SAN（备用域名）、密钥用途、CRL/OCSP 吊销检查地址等。 |

### 二、信任锚：根证书（Root Certificate）

- **本质**：**自签名证书**（Issuer = Subject），由 CA 机构用自己的私钥签名。
- **存储位置**：操作系统（Windows 证书管理器）或浏览器（Firefox 独立证书库）内置的**根证书库**。
- **信任模型**：**无条件信任**。只要链条顶端指向一个内置根证书，整个链就可信。

### 三、证书链的构成

通常为 **三级结构**（部分场景为两级或四级）：

```text
[ 根证书 (Root CA) ]            ← 内置信任，离线保存，私钥绝对安全
        │ (用根私钥签名)
        ▼
[ 中间证书 (Intermediate CA) ]  ← 由根授权签发，负责签发下级证书
        │ (用中间证书私钥签名)
        ▼
[ 服务器证书 (Leaf Certificate) ] ← 部署在 Nginx/Apache，包含网站域名
```

**为什么需要中间证书？**
- **安全隔离**：根证书私钥离线存储，极少使用。中间证书私钥在线，即使泄露也只影响其下发的部分证书，可通过吊销中间证书止损，避免动摇整个 PKI 体系根基。

### 四、验证逻辑的完整步骤（逐层向上验证）

当浏览器收到服务器发送的证书链（至少包含服务器证书，通常附带中间证书）后，执行以下验证：

#### 步骤 1：链的构建
- 浏览器从服务器证书的 `Issuer` 字段找到上级 CA 名称。
- 在服务器发送的证书包里或本地缓存中寻找匹配的中间证书。
- 重复此过程，直到找到一个 `Issuer` 存在于系统根证书库中的证书。

#### 步骤 2：逐级签名验证（核心）
对于链上的每一对 **（下级证书，上级证书）**：
1. 取出上级证书中的 **公钥**。
2. 用该公钥解密下级证书的 **数字签名**，得到一个 Hash 值 `Hash1`。
3. 浏览器用相同的 Hash 算法计算下级证书（除去签名部分）的内容摘要，得到 `Hash2`。
4. 比较 `Hash1 == Hash2`。
   - **相等**：说明下级证书确实是由持有上级证书私钥的机构签发，内容未被篡改。
   - **不相等**：证书链断裂，浏览器报错 `NET::ERR_CERT_AUTHORITY_INVALID`。

#### 步骤 3：有效期检查
- 验证链上每一张证书的 `Not Before` ≤ 当前时间 ≤ `Not After`。
- 任何一张过期，浏览器报错 `NET::ERR_CERT_DATE_INVALID`。

#### 步骤 4：域名匹配检查
- 检查服务器证书的 **Subject CN**（已废弃但仍兼容）或 **Subject Alternative Name (SAN)** 扩展项，是否包含用户访问的**确切域名**。
- 若 `https://www.example.com` 的证书只签发给 `*.example.net`，浏览器报错 `NET::ERR_CERT_COMMON_NAME_INVALID`。

#### 步骤 5：吊销状态检查（可选但重要）
- **CRL (Certificate Revocation List)**：浏览器下载 CA 发布的吊销列表文件，检查证书序列号是否在列。
- **OCSP (Online Certificate Status Protocol)**：浏览器向 CA 的 OCSP 响应服务器实时查询证书序列号状态。
- **OCSP Stapling**：服务器主动定时获取 OCSP 响应并附加在 TLS 握手中，减轻客户端负担并保护隐私。

#### 步骤 6：密钥用途与扩展约束检查
- 检查服务器证书的 `Key Usage` 是否包含 `Digital Signature` 或 `Key Encipherment`。
- 检查 `Extended Key Usage` 是否包含 `Server Authentication`。
- 检查中间证书的 `Basic Constraints` 是否标记为 `CA: TRUE`，路径长度约束（`pathlen`）是否允许继续签发下级证书。

### 五、常见失败场景与原因分析

| 报错现象 | 根本原因 | 排查方向 |
| :--- | :--- | :--- |
| `ERR_CERT_AUTHORITY_INVALID` | 浏览器找不到信任的根证书 | 1. 自签名证书未导入系统。<br>2. 服务器**未发送中间证书**，导致链中断。 |
| `ERR_CERT_DATE_INVALID` | 系统时间错误或证书过期 | 检查本地时间，更新证书。 |
| `ERR_CERT_COMMON_NAME_INVALID` | 访问域名不在证书 SAN 列表中 | 证书签发的域名与实际访问域名不匹配。 |
| `NET::ERR_CERT_WEAK_SIGNATURE_ALGORITHM` | 证书使用 SHA-1 等已废弃的签名算法 | 重新签发使用 SHA-256 的证书。 |
| `SSL证书错误：无法验证证书签名` | 中间证书与服务器证书不匹配 | 确认服务器配置的证书链文件顺序正确（服务器证书在上，中间证书在下）。 |

### 六、面试追问：中间证书不发送会怎样？

**回答要点**：
- 绝大多数浏览器（Chrome、Edge）会尝试通过 **AIA（Authority Information Access）扩展** 中记录的 URL 自动下载缺失的中间证书，完成链构建，用户可能无感知。
- 但部分客户端（如 Java 原生 `HttpURLConnection`、Python `requests` 未开启 certifi）不会自动下载，会导致**连接失败**。
- **最佳实践**：服务器必须配置**完整的证书链**（Leaf + Intermediate），确保所有客户端都能成功验证。

```text

        ▼

        │
        ▼

        ▼
浏览器: 用中间证书公钥验证服务器证书签名？ ─── 否 ──→ 报错签名无效
        │ 是
        ▼
浏览器: 检查服务器证书有效期？ ─── 否 ──→ 报错证书过期
        │ 是
        ▼
浏览器: 中间证书的 Issuer 在根证书库中？
        │ 是
        ▼
浏览器: 用根证书公钥验证中间证书签名？ ─── 否 ──→ 报错链断裂
        │ 是
        ▼
浏览器: (可选) 查询 OCSP/CRL 吊销状态
        │
        ▼
    验证通过，建立 TLS 连接
```

### 四.9 递归查询 vs 迭代查询。 DNS 劫持与 HTTPDNS 解决方案
DNS 是互联网的**地址簿**，将人类可读的域名（如 `www.example.com`）转换为机器可路由的 IP 地址（如 `93.184.216.34`）。它的查询方式分为**递归查询**和**迭代查询**，而传统的 DNS 协议存在**劫持风险**，由此催生了 **HTTPDNS** 解决方案。

### 一、递归查询 vs 迭代查询

#### 1. 递归查询
**一句话定义**：客户端将解析任务**全权委托**给本地 DNS 服务器，由本地 DNS 服务器负责最终返回结果，期间可能多次发起请求。

```text
客户端 → 本地 DNS 服务器：帮我查 www.example.com，我等你结果。
本地 DNS 服务器：好的，我替你跑腿，查不到不罢休。
```

**工作流程**：
1. 客户端向本地 DNS（如 `8.8.8.8`）发起查询。
2. 本地 DNS **代替客户端**逐级向上查询，直到获得最终 IP。
3. 本地 DNS 将结果返回给客户端。

**特点**：
- 客户端**只发一次请求**，等待即可。
- 对本地 DNS 压力大，但它可以**缓存**结果加速后续查询。
- 家庭路由器、公司内网 DNS 默认采用此模式。

#### 2. 迭代查询
**一句话定义**：本地 DNS 服务器作为“向导”，每次只告诉客户端**下一步该问谁**，由客户端自己逐级追问。

```text
客户端 → 根 DNS：www.example.com 的 IP 是啥？
根 DNS：我不知道，但 .com 的服务器是 192.0.2.1，你去问它。
客户端 → .com DNS：www.example.com 的 IP 是啥？
.com DNS：我不知道，但 example.com 的权威 DNS 是 203.0.113.5，你去问它。
客户端 → 权威 DNS：www.example.com 的 IP 是啥？
权威 DNS：是 93.184.216.34。
```

**特点**：
- 客户端需要**多次发起请求**，但每一步都得到明确指引。
- 减轻了根服务器的负担（它只负责指路，不负责跑腿）。

#### 3. 实际混合模型
在实际互联网中，客户端与本地 DNS 之间通常使用**递归查询**，而本地 DNS 向上级（根、顶级域、权威 DNS）查询时使用**迭代查询**。这种分工兼顾了用户体验和系统可扩展性。

### 二、DNS 劫持

#### 1. 什么是 DNS 劫持？
**定义**：攻击者通过篡改 DNS 解析过程，将用户对合法域名的访问**重定向到恶意 IP**，实现钓鱼、弹窗广告或流量窃取。

**常见劫持手段**：
| 劫持类型 | 攻击位置 | 原理 |
| :--- | :--- | :--- |
| **本地 Hosts 劫持** | 用户电脑/手机 | 修改 `hosts` 文件，强制将域名指向错误 IP。 |
| **路由器劫持** | 家庭/企业路由器 | 篡改路由器 DNS 设置，将所有设备流量导向恶意 DNS。 |
| **中间人劫持** | ISP 或公共 Wi-Fi | 运营商/攻击者监听 UDP 53 端口的 DNS 请求，抢先返回虚假应答。 |
| **缓存投毒** | 递归 DNS 服务器 | 向 DNS 服务器注入伪造的解析记录，污染其缓存。 |

#### 2. 典型现象
- 访问淘宝却跳转到钓鱼网站。
- 网页右下角频繁出现无法关闭的运营商广告弹窗。
- 明明输入正确网址，却提示“网站证书错误”。

### 三、HTTPDNS 解决方案

#### 1. 传统 DNS 的致命弱点
- **明文传输**：基于 UDP 的 DNS 查询无加密，中间人可轻易篡改响应。
- **运营商劫持**：部分 ISP 为了推送广告或节省跨网结算费用，会主动篡改 DNS 结果。
- **LocalDNS 出口单一**：移动端使用运营商 LocalDNS，可能因跨地域、跨运营商导致解析 IP 不是最优，访问延迟高。

#### 2. HTTPDNS 原理
**核心思想**：**绕过运营商 LocalDNS**，客户端直接通过 **HTTP/HTTPS 协议** 向可信的 HTTPDNS 服务商（如阿里云、腾讯云、DNSPod）发起域名解析请求。

```text
[ 传统 DNS 路径 ]
App → 系统 DNS 解析器 → 运营商 LocalDNS → 可能被劫持 → 返回 IP

[ HTTPDNS 路径 ]
App → HTTPS 请求 → HTTPDNS 服务端 → 权威 DNS → 返回最优 IP
         ↑
    加密、鉴权、无劫持
```

#### 3. HTTPDNS 的核心优势
| 优势 | 说明 |
| :--- | :--- |
| **防劫持** | 全程 HTTPS 加密，中间人无法篡改响应内容。 |
| **精准调度** | HTTPDNS 服务端根据用户真实出口 IP 返回**同运营商、同地域**的最优节点 IP，避免跨网访问。 |
| **快速生效** | 绕过 LocalDNS 缓存，域名变更（如故障切换）秒级生效。 |
| **扩展性强** | 可在请求中携带用户标识、设备信息，实现更精细的灰度调度。 |

#### 4. 落地实践
- **客户端集成**：App 内嵌 HTTPDNS SDK，拦截系统 `getaddrinfo` 调用，改为 HTTPDNS 请求。
- **兜底机制**：若 HTTPDNS 请求超时或失败，自动降级至系统 DNS 解析，确保可用性。
- **缓存策略**：解析结果在本地缓存，并设置合理的 TTL，避免频繁请求。

### 四、总结对比

| 对比维度 | 传统 DNS | HTTPDNS |
| :--- | :--- | :--- |
| **查询方式** | 递归 / 迭代 | HTTP/HTTPS API 调用 |
| **加密性** | 明文 UDP | HTTPS 加密 |
| **劫持风险** | 高（运营商、路由器、中间人） | 极低（证书校验 + 加密信道） |
| **调度精度** | 依赖 LocalDNS 出口 IP，可能不准 | 基于客户端真实 IP，精准就近接入 |
| **生效速度** | 受 TTL 和 LocalDNS 缓存影响，分钟级 | 可主动刷新缓存，秒级生效 |
| **典型应用** | 网页浏览、传统 PC 软件 | 移动 App、对可用性要求高的 API 服务 |
